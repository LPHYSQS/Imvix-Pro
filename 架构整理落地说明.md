# 架构整理落地说明

## 文档用途

这份文档不是“重构完成报告”，而是给下一任继续推进时使用的交接说明。

目标只有三个：

- 说明目前已经推进到了哪里
- 说明这一阶段刻意没有动什么
- 说明下一步最应该从哪个小切口继续往前推

请继续保持“小步推进、边改边验证、避免一次性大拆”的工作方式。

---

## 一页速览

### 目前已经立起来的边界

- `ApplicationPreferences`：应用级默认偏好
- `ConversionJobDefinition`：当前任务 / 冻结任务的定义
- `WatchProfile`：监听模式配置
- `PreviewToolState` / `PreviewSessionState`：预览工具状态与会话上下文
- `ConversionPlan` / `ConversionRuleSummary`：执行前的轻量规则规划与共享规则摘要出口

### 目前已经落地的关键方向

- 监听模式不再长期直接吃主界面的瞬时状态，而是优先依赖冻结快照
- AI 抠图已经从批量转换 / 预设语义里逐步抽离，回归预览工具语义
- `ConversionPlanningService` 已经开始承担“规则显式化”的职责
- `AI 适用性 / 跳过 AI / 监听模式附加规则` 已经开始共享同一份规则摘要，而不是继续在多个出口各自手拼
- 右侧预览区已经有了“本次任务分析”这类明确的规则出口
- 体积预估已经补上 AI / 多页 PDF / 多帧 GIF 的免责声明
- 透明背景被压平成背景填充的规则，已经开始从 `ViewModel` 临时判断收口到 `ConversionPlan`

### 当前仍然故意没有动的高风险区

- 没有直接大拆 `MainWindowViewModel`
- 没有直接大拆 `ImagePreviewWindow`
- 没有全面移除 `AppServices`
- 没有重写整个转换执行主链
- 没有一次性删除旧模型上的全部历史字段

一句话总结当前状态：

> 现在还不是“已经整理完”，而是“已经把正确的推进边界和主方向打出来了”。

## 到目前为止已经推进了什么

下面改成压缩归纳版，只保留真正对后续接手有帮助的主线。

### 已经完成的阶段性收口

- 状态边界已经立起来：`ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState`、`PreviewSessionState`、`ConversionPlan` 已经不是概念草图，而是实际开始承接职责。
- 旧设置兼容层已经立起来：`Services/AppSettingsStateMapper.cs` 已经把旧设置对象与新状态模型之间的映射收口，后续迁移不需要再回到“大而全”的旧对象上直接扩字段。
- 监听模式已经从“主界面瞬时状态的附属触发器”往“长期运行任务”推进：启用监听时固化 `ConversionJobDefinition` 快照，监听执行优先吃冻结规则，也支持显式“保存当前参数为监听配置”。
- AI 抠图已经开始回归预览工具语义：相关状态从批量转换 / 预设语义里逐步抽离，避免继续和主批处理管线混用。
- 轻量规划层已经开始接管解释性规则：`ConversionPlanningService` 已承接 AI 是否生效、哪些输入跳过 AI、GIF/PDF 展开、预计输出项、工作项、体积预估免责声明等说明性信息。
- 透明被压平为背景填充的规则，已经不再只是 `ViewModel` 里的局部判断，而是开始收口到 `ConversionPlan`，并被转换前预警、右侧任务分析、当前选中文件告警共同消费。

### 我这一轮新增推进

- 新增了共享规则摘要对象：`ConversionRuleSummary`，把 AI 适用性、跳过 AI、GIF/PDF 展开、背景填充这几类规则，从“散落布尔值 + 临时字符串”推进到可复用的语义对象。
- `ConversionPlanningService` 现在不只会产出 `ConversionPlan`，还会显式产出批处理规则摘要、监听规则摘要，并统一承接监听模式对 GIF / PDF 的内容感知附加规则。
- 手动转换前预警、右侧“本次任务分析”、监听配置摘要，这三个出口现在已经不再各自重写 AI / GIF / PDF 规则说明，而是开始消费同一层规则语义。
- `MainWindowViewModel` 中与这组规则相关的共享文案组装，已经单独收口到 `MainWindowViewModel.RuleSummary.cs`；这一块后续应继续收，不要重新把同类字符串手拼散回 `ConversionViewModel` / `ConversionWatchViewModel`。
- 为这次新增的通用 AI 回退说明补了 10 种语言文案；后续凡是再新增 UI 文案键，继续保持这 10 种语言同步。
- 增加了轻量回归验证：`Tests/ImvixPro.Tests/ImageConversionTests.cs` 已补上监听规则摘要的稳定性测试。

### 当前仍未解掉的核心堵点

- `AI 适用性 / 跳过 AI / 监听模式附加规则` 这一包已经开始共用规则摘要，但高压缩、GIF 转 PDF 大帧风险、PDF 锁定跳过、当前文件告警等诊断逻辑仍散落在 `ConversionViewModel`。
- `ActiveWarnings`、转换前确认弹窗、右侧任务分析、完成后摘要这些出口，仍没有共享同一份更完整的 diagnostics summary。
- `MainWindowViewModel` 和 `ImagePreviewWindow` 仍然是两个大宿主，边界虽然开始清楚，但结构风险还在。
- `AppServices` 仍然是运行期跨层入口，依赖关系还不够透明。
- `AppSettings` / `ConversionOptions` / `ConversionPreset` 仍然有兼容层噪音，不能再继续横向长字段。

## 当前刻意没有做的事

为了控制风险，下面这些事情仍然是有意识地先不硬拆：

- 不直接大拆 `MainWindowViewModel` partial 体系。
- 不直接大拆 `ImagePreviewWindow` 工具宿主。
- 不直接重写转换执行主链。
- 不直接把 `AppServices` 一次性拿掉。
- 不为了“看起来干净”一次性删除旧模型所有历史字段。
- 不引入新的测试工程或大批一次性验证脚本。

原因不变：

> 如果在规则层还没进一步收稳的时候，同时叠加结构拆分、依赖迁移、执行器重写，风险会急剧上升。

## 下一任最建议从哪里继续推

### 先说结论

下一任不要再做“一个文案点 / 一个局部方法 / 一个提示出口”这种过小切口了。

更合适的粒度应该是：

- 一次拿下一整类同源规则。
- 一次至少收掉这类规则在 2 到 3 个出口上的重复实现。
- 一次做完之后，要能真实减少后续轮次，而不是只让文档里的“还要 6 到 8 次”保持不变。

### 下一任最推荐的中等偏重工作包

建议下一任直接做一个“诊断出口统一包”，范围比这一轮再重一点，但仍然是可控的中等偏重范围，不是失控式重构。

这一包建议至少包含下面四件事：

1. 在现有 `ConversionRuleSummary` 的基础上，再往前推一层 `DiagnosticsSummary` 或等价对象，把下面几类“仍然留在 ViewModel 的诊断逻辑”一起收口：
- 高压缩风险
- GIF 转 PDF 大帧风险
- PDF 锁定后跳过
- 当前选中文件的背景填充告警
- 体积预估免责声明
2. 让下面至少三个出口尽量消费同一份 diagnostics 结果，而不是继续各自手拼：
- 转换前确认弹窗
- 右侧“本次任务分析” / 当前文件告警
- `ActiveWarnings` 或等价的运行前提示出口
3. 收口后顺手删掉 `ConversionViewModel` 中剩余的局部 if/字符串拼装，不要让这一轮已经收进去的规则又被新的分支绕回去。
4. 如果时间还够，再开始收敛“一次性完成摘要”的唯一出口，减少底部状态、完成摘要、历史卡片之间对同一任务信息的重复表达。

### 为什么我建议下一任把范围加大

- 现在剩下的这些逻辑，本质上已经不是孤立问题，而是一组“同源规则在多个出口重复实现”的问题。
- 如果继续按“一轮只修一个很小点”的方式推进，剩余轮次不会明显下降。
- 反过来，只要一次吃掉一个“规则包”，哪怕工作量更重一点，整体推进速度会明显好于继续零切口推进。
- 这一建议仍然没有越线到“大拆主界面”或“重写执行器”，所以风险仍在可控范围内。

### 建议的实际落地顺序

- 第一步：盘点 `ConversionViewModel` 里还剩哪些 warning / disclaimer / 当前文件告警仍是纯手拼。
- 第二步：在规划层或 diagnostics 层扩展统一摘要对象，不要继续只返回零散字符串或散落布尔值。
- 第三步：优先替换转换前确认、右侧任务分析、`ActiveWarnings` 三处出口，再删除旧分支。
- 第四步：完成后必须做 `build + 启动 + 生成残留清理`。

### 如果下一任还有余力

- 可以开始小幅处理“重复状态出口”问题，优先收敛一次性完成摘要出口，不要再让多个区域同时讲同一件事。
- 但下一轮仍然不建议直接大拆 `MainWindowViewModel`，除非前面的 diagnostics 出口统一已经完成并且验证稳定。

## 下一任暂时不建议马上做的事

- 不建议直接全面拆 `MainWindowViewModel`
- 不建议直接全面迁移 `AppServices`
- 不建议一边重写执行主链、一边重做预览窗口、一边收规则层

当前更好的策略是：

> 先用一个中等范围工作包，把同源规则的多个出口一起收掉，再考虑更大的结构动作。

## 一个实用判断标准

如果你新增或调整一条规则时，出现了下面任一情况：

- 需要同时改很多个 `ViewModel` 条件分支
- 需要同时补很多个 UI 入口说明
- 需要猜“手动转换”和“监听模式”是否表现一致
- 需要在预警、摘要、提示里重复写相同逻辑

那基本说明这条规则还没有真正进入 `ConversionPlan / WatchProfile / Policy` 这一层。

这时正确动作通常不是继续堆分支，而是先收口，然后顺手删掉旧出口。

- 不要再按“一个极小点一轮”推进，改成“一个中等范围规则包一轮”。
- 先做“边界更清楚、规则出口更统一”，再做“大结构更漂亮”。
- 不新增无谓的测试垃圾、临时脚本和一次性验证残留。
- 收尾时清理生成产物，但不要误删真实测试源码和夹具。
- 每次改完至少保证 `build` 通过、主程序可以正常启动。

这份工作到现在，已经不适合再按“一个提醒文案一轮”去推进了。
