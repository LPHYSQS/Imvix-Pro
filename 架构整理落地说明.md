### 目前已经立住的核心边界

- `ApplicationPreferences`：应用级默认偏好
- `ConversionJobDefinition`：当前任务 / 冻结任务定义
- `WatchProfile`：监听模式配置
- `PreviewToolState` / `PreviewSessionState`：预览工具状态与预览会话上下文
- `PreviewSelectionSnapshot` / `PreviewSelectionState`：当前选中文件预览子域
- `ConversionPlan` / `ConversionRuleSummary` / `ConversionDiagnosticsSummary`：执行前规划、规则摘要、诊断摘要
- `RuntimeStatusSummary` / `CompletionSummaryModel` / `WatchRuntimeStatusSummary`：运行态与完成态共享摘要语义
- `ConversionSummaryCoordinator`：完成摘要、历史档案、完成弹窗的统一协调层
- `HistoryState` / `NotificationState` / `TaskAnalysisState`：历史、通知、任务分析三个独立结果子域
- `PreviewRenderCoordinator`：主窗口预览切换时序协调
- `MainWindowShellCoordinator`：主窗口宿主层协调
- `MainWindowHostController`：tray、window placement、startup path、folder picker 等宿主杂务
- `MainWindowConfigurationCoordinator` / `MainWindowConfigurationSnapshot`：当前任务 / 默认偏好 / 预览工具 / 监听档案的配置组装边界

## 这一次收官新增完成的内容

### 1. `ImagePreviewWindow` 已经真正进入 tool-host 形态

- 预览窗口不再只是“先落一刀条码 controller”的起步状态
- 现在已经有 `PreviewOcrToolController`、`PreviewQrToolController`、`PreviewBarcodeToolController` 三个真实 tool controller
- OCR / QR / 条码三条识别链路已经统一走共享的 recognition session 流程：取消旧任务、置忙、展开面板、执行识别、回收状态不再三份平行增长
- 结果整形已经从窗口主体里进一步抽离，窗口本体只保留 UI 呈现与宿主调度职责

### 2. `MainWindowViewModel` 的配置组装边界已经进一步收口

- `BuildCurrentConversionOptions()`
- `BuildCurrentJobDefinition()`
- `BuildCurrentPreviewToolState()`
- `BuildConfiguredWatchProfile()`
- `BuildPreset()`

上面这些原先散落在 `MainWindowViewModel` 里的配置拼装逻辑，现在都已经退回到 `MainWindowConfigurationCoordinator`。

当前主窗口通过 `MainWindowConfigurationSnapshot` 先显式收集当前状态，再交给 coordinator 生成：

- 当前任务 `ConversionOptions`
- 当前任务定义 `ConversionJobDefinition`
- 预览工具状态 `PreviewToolState`
- 预览会话状态 `PreviewSessionState`
- 监听档案 `WatchProfile`
- 应用偏好 `ApplicationPreferences`
- 预设 `ConversionPreset`

这意味着“当前任务 / 默认偏好 / 预览工具 / 监听档案”这四类语义，已经不再继续直接混在主 ViewModel 的局部临时拼装里。

### 3. `AppServices` 在 main-window / preview path 上的运行期直连已经再压缩一轮

- `FileListViewModel` 不再直接调用 `AppServices.Logger`
- `ImagePreviewWindow` 不再直接 new OCR / QR service 再自己决定识别链路，改为走 services factory 注入的 tool controller
- 当前 main-window / preview path 上，`AppServices` 基本只剩构造入口与装配职责，运行期主要逻辑已经退回到明确的 services / coordinator / controller 边界

### 4. 第二大块要求的“字段归属再收紧一轮”已经完成

这次没有去做危险的“一次性删旧字段大迁移”，但已经把最常走的一条路径理顺：

- 当前任务配置通过 `MainWindowConfigurationSnapshot -> ConversionOptions`
- 默认偏好通过 `MainWindowConfigurationSnapshot -> ApplicationPreferences`
- 预览工具配置通过 `MainWindowConfigurationSnapshot -> PreviewToolState`
- 监听模式配置通过 `MainWindowConfigurationSnapshot -> WatchProfile`

也就是说，虽然兼容层仍然存在，但新结构已经不再鼓励继续横向往 `AppSettings` / `ConversionOptions` / `ConversionPreset` 里无脑追加混合字段。

### 5. 两大块已经重新合并回一份完整结论

这一阶段最初拆开的两大块现在已经都做完：

#### 第一大块：`Preview/Host Consolidation 半场收口包`

已完成的核心点：

1. `PreviewRenderCoordinator` 收住了主窗口 GIF / PDF / PSD 预览切换时序
2. `MainWindowHostController` 接手了主窗口宿主杂务
3. `PreviewSelectionState`、`TaskAnalysisState`、`HistoryState`、`NotificationState` 已经各自立住
4. `MainWindowShellCoordinator` 已经接手 completion dialog、tray、预览窗口 / 文件详情窗口等宿主协调入口

#### 第二大块：`Composition/Tool Host Finalization 终局收口包`

已完成的核心点：

1. `ImagePreviewWindow` 已经成为 3 个真实 tool controller 的宿主
2. `MainWindowConfigurationCoordinator` 已经接手主窗口最常走的配置组装边界
3. `AppServices` 在主窗口 / 预览窗口运行期路径上的直连已经进一步压缩
4. 当前任务 / 默认偏好 / 预览工具 / 监听档案这四类语义，已经通过 snapshot + coordinator 的方式重新分层

到这里为止，这两大块已经不再是“一个已完成、一个待推进”，而是已经合并成为同一轮架构整理的完整收官结果。

## 后续维护者现在应该怎么理解这个仓库

### 不要再回头重开这两大块

这两大块已经收完，后续不要再把工作描述成：

- “先回头补一下 Preview/Host Consolidation 的尾巴”
- “还剩 Composition/Tool Host Finalization 最后一半”

这两种说法现在都已经过期。

### 后续如果继续做功能，请直接沿着现有边界推进

#### 如果新增预览工具

- 不要继续把工具状态、忙碌状态、结果整形直接堆回 `ImagePreviewWindow`
- 优先新增独立 `Preview*ToolController`
- 预览窗口只负责会话调度、互斥控制和 UI 呈现

#### 如果新增当前任务 / 默认设置 / 监听模式相关字段

- 不要先往 `AppSettings` / `ConversionOptions` / `ConversionPreset` 三个对象里横向加字段再说
- 先判断它究竟属于：
  - `ApplicationPreferences`
  - `ConversionJobDefinition`
  - `PreviewToolState`
  - `WatchProfile`
- 再决定要不要进入 `MainWindowConfigurationSnapshot` 和 `MainWindowConfigurationCoordinator`

#### 如果新增主窗口宿主级行为

- 先看它属于 `MainWindowShellCoordinator` 还是 `MainWindowHostController`
- 不要重新把 tray / placement / picker / startup 这类杂务塞回 `MainWindow.axaml.cs`

### 不建议现在去做的事

- 不建议为了“更彻底”一次性删空所有旧兼容字段
- 不建议为了“更漂亮”马上把所有 partial 再拆成很多类
- 不建议一边重写执行主链、一边重做主窗口、一边改预览工具宿主

当前更好的策略是：

> 在已经立住的边界上继续演进功能，而不是重新打开这一轮已经收官的结构整顿工程。
