# 架构整理落地说明

## 文档用途

这份文档不是“重构完成报告”，而是给下一任继续推进时使用的交接说明。

目标只有三个：

- 说明目前已经推进到了哪里
- 说明这一阶段刻意没有动什么
- 说明下一步最应该从哪个小切口继续往前推

请继续保持“小步推进、边改边验证、避免一次性大拆”的工作方式。

## 一页速览

### 目前已经立起来的边界

- `ApplicationPreferences`：应用级默认偏好
- `ConversionJobDefinition`：当前任务 / 冻结任务的定义
- `WatchProfile`：监听模式配置
- `PreviewToolState` / `PreviewSessionState`：预览工具状态与会话上下文
- `ConversionPlan` / `ConversionRuleSummary` / `ConversionDiagnosticsSummary`：执行前的轻量规划、共享规则摘要、共享诊断摘要

### 目前已经落地的关键方向

- 监听模式不再长期直接吃主界面的瞬时状态，而是优先依赖冻结快照
- AI 抠图已经从批量转换 / 预设语义里逐步抽离，回归预览工具语义
- `ConversionPlanningService` 已经开始承担“规则显式化”的职责
- `AI 适用性 / 跳过 AI / 监听模式附加规则` 已经开始共享同一份规则摘要，而不是继续在多个出口各自手拼
- `高压缩 / GIF 转 PDF 大帧 / PDF 锁定跳过 / 体积预估免责声明` 已经开始共享同一层 diagnostics 语义，而不是继续散落在 `ConversionViewModel`
- `RuntimeStatusSummary` / `CompletionSummaryModel` / `ConversionStatusSummaryService` 已经开始承担“运行中状态 + 完成摘要 + 历史摘要”的共享组装职责
- 右侧预览区已经有了“本次任务分析”这类明确的规则出口
- 预览区已经不再直接复用任务级 `CurrentFile`，而是回到“当前选中文件”语义
- 体积预估已经补上 AI / 多页 PDF / 多帧 GIF 的免责声明
- 透明背景被压平成背景填充的规则，已经开始从 `ViewModel` 临时判断收口到 `ConversionPlan`

### 当前仍然故意没有动的高风险区

- 没有直接大拆 `MainWindowViewModel`
- 没有直接大拆 `ImagePreviewWindow`
- 没有全面移除 `AppServices`
- 没有重写整个转换执行主链
- 没有一次性删除旧模型上的全部历史字段

一句话总结当前状态：

> 现在还不是“已经整理完”，而是“已经把正确的推进边界和主方向打出来了”。

## 到目前为止已经推进了什么

下面改成压缩归纳版，只保留真正对后续接手有帮助的主线。

### 已经完成的阶段性收口

- 状态边界已经立起来：`ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState`、`PreviewSessionState`、`ConversionPlan` 已经不是概念草图，而是实际开始承接职责。
- 旧设置兼容层已经立起来：`Services/AppSettingsStateMapper.cs` 已经把旧设置对象与新状态模型之间的映射收口，后续迁移不需要再回到“大而全”的旧对象上直接扩字段。
- 监听模式已经从“主界面瞬时状态的附属触发器”往“长期运行任务”推进：启用监听时固化 `ConversionJobDefinition` 快照，监听执行优先吃冻结规则，也支持显式“保存当前参数为监听配置”。
- AI 抠图已经开始回归预览工具语义：相关状态从批量转换 / 预设语义里逐步抽离，避免继续和主批处理管线混用。
- 轻量规划层已经开始接管解释性规则：`ConversionPlanningService` 已承接 AI 是否生效、哪些输入跳过 AI、GIF/PDF 展开、预计输出项、工作项、体积预估免责声明等说明性信息。
- 透明被压平为背景填充的规则，已经不再只是 `ViewModel` 里的局部判断，而是开始收口到 `ConversionPlan`，并被转换前预警、右侧任务分析、当前选中文件告警共同消费。

### 我这一轮新增推进

- 已新增 `RuntimeStatusSummary` 与 `CompletionSummaryModel`，把“运行中状态”和“完成后摘要”从零散字符串拼装，开始收进共享模型层。
- 已新增 `Services/ConversionStatusSummaryService.cs`，把手动转换进度、完成摘要、历史摘要、监听历史落盘前的共享组装逻辑收口到一个入口，不再继续让 `ConversionViewModel` 和历史组装各自手拼。
- 底部状态栏现在回到“只表达运行中状态”的职责：完成后不再把整批任务摘要塞回 `CurrentFile`，`CurrentFile` 不再兼任“当前文件名”和“任务完成摘要文本”两种语义。
- 完成弹窗与历史卡片现在已经开始消费同一份 `CompletionSummaryModel` 语义；监听模式写入历史时也已经接上同一套完成摘要模型，而不是继续走单独的历史拼装分支。
- 预览区已经不再直接消费任务级 `CurrentFile`，而是改成消费当前选中文件名，避免预览区继续承担任务状态摘要出口。
- 已把测试可见性显式化：新增 `Properties/AssemblyInfo.cs` 的 `InternalsVisibleTo("ImvixPro.Tests")`，让现有测试对内部导入工厂的依赖不再是隐式碰运气状态。

- 已补回归测试：`Tests/ImvixPro.Tests/ImageConversionTests.cs` 现在额外覆盖 runtime/completion 摘要对象的稳定结构，不只是 diagnostics。

### 当前仍未解掉的核心堵点

- 现在手动转换的 runtime/completion 已经开始统一，但监听模式运行中的 `WatchStatusText / WatchMetricsText / WatchCurrentFile` 仍是另一套旁路语义，还没有真正落到共享 runtime summary 上。
- 完成弹窗、历史卡片虽然已经共用 `CompletionSummaryModel`，但“完成摘要展示方式”仍然散在 `MainWindowViewModel` 里，没有独立 coordinator / presenter。
- `MainWindowViewModel` 和 `ImagePreviewWindow` 仍然是两个大宿主，边界虽然开始清楚，但结构风险还在。
- `AppServices` 仍然是运行期跨层入口，依赖关系还不够透明。
- `AppSettings` / `ConversionOptions` / `ConversionPreset` 仍然有兼容层噪音，不能再继续横向长字段。

## 下一任最建议从哪里继续推

### 先说结论

下一任不要再回到“一个文案点 / 一个局部方法 / 一个提示出口”这种过小切口了，也不要重复再做一轮 diagnostics 收口的尾巴活。

更合适的粒度应该是：

- 一次拿下一整类同源规则。
- 一次至少收掉这类规则在 2 到 3 个出口上的重复实现。
- 一次做完之后，要能真实减少后续轮次，而不是只让文档里的“还要 4到 5 次”保持不变。

### 下一任最推荐的中等偏重工作包

建议下一任不要再重复做“手动转换摘要”的补丁活了，直接上一个“监听运行状态与摘要协调包”，范围要比这一轮再重一点，但仍然控制在“中等偏重、非失控式重构”的范围内。

这一包建议至少包含下面四件事：

1. 把 `WatchStatusText / WatchMetricsText / WatchCurrentFile / IsWatchProcessing` 收口到共享 `RuntimeStatusSummary` 或等价 watch runtime summary，对监听模式建立正式的运行中摘要对象。
2. 让监听模式、手动模式至少在下面两个维度上共用一套摘要语义，而不是继续双轨：
- 运行中状态
- 完成后/失败后摘要入口
3. 把完成摘要相关格式化逻辑继续从 `MainWindowViewModel` 里往 summary presenter / coordinator 抽一层，避免 `BuildConversionSummaryText`、历史卡片组装、监听状态提示继续粘在主 ViewModel 里。
4. 在做完上面三件事后，再顺手清理掉监听路径里剩余的局部字符串拼装和旧字段依赖，确保新增规则不再先改 watch 分支、再改 manual 分支。

### 为什么我建议下一任把范围加大

- 手动转换这条线现在已经把 runtime/completion 模型立起来了，真正还在拉低推进效率的，已经变成监听模式继续走旁路状态语义。
- 如果下一任还只修一个 watch 文案点或一个局部状态分支，剩余轮次还是降不下来。
- 反过来，只要下一任一次吃掉“监听运行状态 + 摘要协调”这一整包，后面再处理 ViewModel 拆分、scheduler、AppServices 时，manual/watch 两条线会明显更容易并轨。
- 这一建议仍然没有越线到“大拆主界面”或“重写执行器”，所以风险仍在可控范围内。

### 建议的实际落地顺序

- 第一步：盘点监听模式里 `WatchStatusText / WatchMetricsText / WatchCurrentFile / IsWatchProcessing` 这几处，到底哪些信息已经和 `RuntimeStatusSummary` 重叠。
- 第二步：先立 watch runtime summary 对象，再让 `RefreshWatchStatus` 改成消费 summary，而不是继续手拼字符串。
- 第三步：把监听成功/失败后的摘要入口也接到统一 completion 语义上，减少 watch/manual 双轨格式化。
- 第四步：完成后必须做 `build + test + 启动 + 生成残留清理`。

### 如果下一任还有余力

- 可以顺手把 `BuildConversionSummaryText`、历史卡片格式化、监听状态提示继续往单独 presenter/coordinator 收，不要让主 ViewModel 再长一层展示逻辑。
- 但下一轮仍然不建议直接大拆 `MainWindowViewModel`，除非前面的 watch runtime/summary 统一已经完成并且验证稳定。

当前更好的策略是：

## 一个实用判断标准

如果你新增或调整一条规则时，出现了下面任一情况：

- 需要同时改很多个 `ViewModel` 条件分支
- 需要同时补很多个 UI 入口说明
- 需要猜“手动转换”和“监听模式”是否表现一致
- 需要在预警、摘要、提示里重复写相同逻辑

那基本说明这条规则还没有真正进入 `ConversionPlan / WatchProfile / Policy` 这一层。

这时正确动作通常不是继续堆分支，而是先收口，然后顺手删掉旧出口。

## 建议继续保持的工作方式

- 不要再按“一个极小点一轮”推进，改成“一个中等范围规则包一轮”。
- 先做“边界更清楚、规则出口更统一”，再做“大结构更漂亮”。
- 不新增无谓的测试垃圾、临时脚本和一次性验证残留。
- 收尾时清理生成产物，但不要误删真实测试源码和夹具。
- 每次改完至少保证 `build` 通过、主程序可以正常启动。

## 给下一任的最后一句话

这份工作到现在，已经不适合再按“一个 watch 文案点一轮”或“一个局部状态分支一轮”去推进了。

请至少拿下一个“监听运行状态与摘要协调包”，让剩余轮次继续真实下降，而不是停在原地。

判断你这轮有没有做对，不是看你加了多少新类，而是看你有没有同时做到下面三件事：

- 多统一一整类 manual/watch 共同摘要出口
- 少留一处 watch/manual 双轨逻辑
- 改完还能稳定 `build + 启动`

只要做到这一点，这个项目就会开始明显从“渐进止血”转向“可持续维护”。
