# 架构整理落地说明

## 文档用途

这份文档不是“重构完成报告”，而是给下一任继续推进时使用的交接说明。

目标只有三个：

- 说明目前已经推进到了哪里
- 说明这一阶段刻意没有动什么
- 说明下一步最应该从哪个中等粒度工作包继续往前推

请继续保持“小步推进、边改边验证、避免一次性大拆”的工作方式，但不要再回到“一次只补一个文案点或一个局部字符串”的推进粒度。

## 一页速览

### 目前已经立起来的边界

- `ApplicationPreferences`：应用级默认偏好
- `ConversionJobDefinition`：当前任务 / 冻结任务的定义
- `WatchProfile`：监听模式配置
- `PreviewToolState` / `PreviewSessionState`：预览工具状态与会话上下文
- `PreviewSelectionSnapshot` / `PreviewSelectionState`：当前选中文件预览、PDF/GIF/PSD 文件级可见性与预览面板显示状态的独立子域
- `ConversionPlan` / `ConversionRuleSummary` / `ConversionDiagnosticsSummary`：执行前的轻量规划、共享规则摘要、共享诊断摘要
- `RuntimeStatusSummary` / `CompletionSummaryModel`：手动与监听共享的运行中 / 完成后摘要语义基础
- `WatchRuntimeStatusSummary`：监听模式正式的运行中摘要对象，不再只靠局部字符串拼装
- `ConversionSummaryCoordinator` / `ConversionSummaryFlowResult` / `ConversionSummaryDialogRequest`：手动与监听共享的“完成后摘要 -> 历史档案 -> 弹窗请求”协调层
- `HistoryState`：历史列表、历史持久化读取、历史卡片展示文本、清空动作的独立子域
- `NotificationState`：失败日志入口、待展示完成弹窗请求的独立子域
- `TaskAnalysisState`：右侧任务分析面板的警告、规则摘要、格式推荐、体积预估与免责声明子域

### 目前已经落地的关键方向

- 监听模式不再长期直接吃主界面的瞬时状态，而是优先依赖冻结快照
- AI 抠图已经从批量转换 / 预设语义里逐步抽离，回归预览工具语义
- `ConversionPlanningService` 已经开始承担“规则显式化”的职责
- `AI 适用性 / 跳过 AI / 监听模式附加规则` 已经开始共享同一份规则摘要，而不是继续在多个出口各自手拼
- `高压缩 / GIF 转 PDF 大帧 / PDF 锁定跳过 / 体积预估免责声明` 已经开始共享同一层 diagnostics 语义，而不是继续散落在 `ConversionViewModel`
- `ConversionTextPresenter` 已开始承担“完成摘要弹窗 / 历史卡片 / 监听状态文本”的共享格式化职责
- `ConversionSummaryCoordinator` 已开始承担“完成后摘要生命周期协调”的职责，manual/watch 不再各自手拼历史项与弹窗文案
- `HistoryState` / `NotificationState` 已经把“完成后遗留状态”从 `MainWindowViewModel` 主体里拆出，主 ViewModel 不再直接持有历史集合与失败日志路径
- `TaskAnalysisState` 已经把“任务分析 + 风险提示 + 格式推荐 + 体积预估”从 `MainWindowViewModel` 的扁平字段里拆出，`RefreshConversionInsights()` 现在改为“生成分析快照 -> 更新子域状态”
- `MainWindow.axaml.cs` 不再订阅手工 completion event，而是改为消费 `NotificationState` 上的待展示弹窗请求
- 右侧预览区已经有了“本次任务分析”这类明确的规则出口
- `PreviewPanel.axaml` 的警告 / 任务分析 / 格式推荐 / 体积预估四块现在已经直接绑定 `TaskAnalysisState`，不再继续吃主 ViewModel 的这组平铺属性
- 预览区已经不再直接复用任务级 `CurrentFile`，而是回到“当前选中文件”语义
- `PreviewSelectionState` 已经开始承接 preview 子域的展示状态，`PreviewPanel.axaml` 的文件预览相关区域不再继续直接绑定一长串主 ViewModel 扁平 preview 属性
- 体积预估已经补上 AI / 多页 PDF / 多帧 GIF 的免责声明
- 透明背景被压平成背景填充的规则，已经开始从 `ViewModel` 临时判断收口到 `ConversionPlan`

### 当前仍然故意没有动的高风险区

- 没有直接大拆 `MainWindowViewModel`
- 没有直接大拆 `ImagePreviewWindow`
- 没有全面移除 `AppServices`
- 没有重写整个转换执行主链
- 没有一次性删除旧模型上的全部历史字段

一句话总结当前状态：

> 现在还不是“已经整理完”，而是“已经把正确的推进边界、共享摘要语义和下一段真正该收口的主线打出来了”。

## 到目前为止已经推进了什么

下面只保留真正对后续接手有帮助的主线，不再展开成大段流水账。

### 已经完成的阶段性收口

- 状态边界已经立起来：`ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState`、`PreviewSessionState`、`ConversionPlan` 已经不是概念草图，而是实际开始承接职责。
- 旧设置兼容层已经立起来：`Services/AppSettingsStateMapper.cs` 已经把旧设置对象与新状态模型之间的映射收口，后续迁移不需要再回到“大而全”的旧对象上直接扩字段。
- 监听模式已经从“主界面瞬时状态的附属触发器”往“长期运行任务”推进：启用监听时固化 `ConversionJobDefinition` 快照，监听执行优先吃冻结规则，也支持显式“保存当前参数为监听配置”。
- AI 抠图已经开始回归预览工具语义：相关状态从批量转换 / 预设语义里逐步抽离，避免继续和主批处理管线混用。
- 轻量规划层已经开始接管解释性规则：`ConversionPlanningService` 已承接 AI 是否生效、哪些输入跳过 AI、GIF/PDF 展开、预计输出项、工作项、体积预估免责声明等说明性信息。
- 透明被压平为背景填充的规则，已经不再只是 `ViewModel` 里的局部判断，而是开始收口到 `ConversionPlan`，并被转换前预警、右侧任务分析、当前选中文件告警共同消费。
- 手动转换已经用 `RuntimeStatusSummary` 和 `CompletionSummaryModel` 收拢“运行中状态 / 完成后摘要 / 历史摘要”的共享语义。

### 我这一轮新增推进

- 已新增 `Models/PreviewSelectionSnapshot.cs` 与 `ViewModels/Main/PreviewSelectionState.cs`，把“当前选中文件预览 + PDF/GIF/PSD 文件级可见性 + 锁定提示 + 页帧指示器”正式收口成独立 feature state。
- `PreviewViewModel.cs` 现在新增 `BuildPreviewSelectionSnapshot()` 与 `RefreshPreviewSelectionState()`，preview 子域第一次拥有了单一的 snapshot -> state 更新入口，而不再只是让 `PreviewPanel.axaml` 直接摊开绑定主 ViewModel 的一长串属性。
- `PreviewPanel.axaml` 本轮已把顶部预览、PDF/GIF/SVG/Icon 区块与 GIF Trim 区块的主要显示绑定切到 `PreviewSelectionState`，主 ViewModel 继续承接编辑值和命令，但 preview 面板对平铺属性的直接耦合已经明显下降。
- 已删掉 `PreviewSelectionFileText` 与 `IsPreviewWindowHintVisible` 这两个最容易继续扩散的 preview 扁平出口，避免出现“新 state 已有、旧提示字段还在继续同步”的双轨。
- 已把 `OnSelectedImageChanged()`、`OnSelectedPreviewChanged()`、PDF/GIF 相关 UI 刷新点与本地化刷新点接入统一 `RefreshPreviewSelectionState()`，确保这次不是简单搬字段，而是正式建立 preview 子域的共享刷新入口。
- 本轮仍然没有新增 UI 文案键，因此没有扩散 10 语种维护面；后续若再新增文案键，仍要同步简中、繁中、英语、日语、韩语、德语、法语、意大利语、俄语、阿拉伯语这 10 种语言。

## 下一任最建议从哪里继续推

### 先说结论

下一任不要再回头补 `history + notification`、`task analysis sidebar`，也不要再围着 `PreviewSelectionState` 做只改几个绑定名的小修了；这些边界现在都已经立住了，继续在这里打转只会重新掉回“边角优化”。

更适合下一任接手的是一个比本轮更重、而且应该明确做成“子域抽离”的工作包：

> **MainWindowShellCoordinator + PreviewRenderCoordinator 收口包**

目标不再是继续证明 preview state 有用，而是顺着这次已经立起来的 `PreviewSelectionState` 往前走，把“窗口宿主协调 + 预览渲染调度”这两块真正还在拖慢推进速度的重区一起拿下来，直接开始削掉 `MainWindow.axaml.cs` 与 `MainWindowViewModel` 的下一整片重量。

### 下一任最推荐的中等偏重工作包

建议下一任至少同时完成下面五件事：

1. 把 `MainWindow.axaml.cs` 里与宿主行为相关的主分支先收口出去，至少覆盖：待展示 completion dialog 消费、Tray hide/restore/exit、BringToFront、预览窗口打开、文件详情窗口打开、PDF 解锁弹窗入口。
2. 让 `PreviewPanel` / 文件列表 / 主窗口不再直接各自决定何时打开预览窗口或详情窗口，而是改成统一的 request/coordinator 入口；code-behind 只负责把平台事件转交给壳层。
3. 顺着这次已经落地的 `PreviewSelectionState`，继续把 `OnSelectedImageChanged()` 里仍然很重的“清理旧预览 -> 恢复 per-file 选择 -> 触发 GIF/PDF/PSD preview load”迁到独立 `PreviewRenderCoordinator` 或等价预览调度层。
4. 上面两步做完后，必须顺手删除 `MainWindow.axaml.cs` 里对应旧分支，以及 `MainWindowViewModel` 里已经可以被 coordinator 替代的 preview 扁平刷新出口，不要留下 coordinator / code-behind / ViewModel 三轨并行。
5. 收尾时必须继续做 `build + test + 启动 + 测试残留清理`，并检查这次没有引入新的文案键；若有新增 UI 文案，仍需同步 10 语种。

### 为什么我建议下一任把范围继续加大

- 这轮已经把 preview 面板的显示状态正式收口到 `PreviewSelectionState`，再回头继续做“再多改几个 preview 绑定”已经不会带来本质收益。
- 现在真正拖慢推进速度的，已经变成“窗口宿主逻辑 + 预览渲染调度”这一整片 still-heavy 的区域；它仍然要求 `MainWindow.axaml.cs` 和 `MainWindowViewModel` 共同维护大量时序分支。
- 只要下一任把 `shell coordination + preview render coordination` 这一整片一起抽出来，后面无论是继续拆 `MainWindowViewModel`，还是整理 `ImagePreviewWindow` 的工具宿主边界，都会一下子轻很多。
- 这个建议虽然比本轮更重，但仍然没有越线到“重写执行主链”或“直接大拆预览窗口”，风险仍是可控的，而且比继续做小补丁更接近真正提速。

### 建议的实际落地顺序

第一步：

- 盘点 `MainWindow.axaml.cs` 里仍然属于宿主协调的全部内容：预览窗口打开、文件详情窗口打开、PDF 解锁、待展示 completion dialog 消费、Tray、窗口可见性、BringToFront、第二实例激活路径。

第二步：

- 先立一个独立 `MainWindowShellCoordinator`（或等价壳层）入口，把上面这些宿主动作统一收口，让 code-behind 先退化成平台事件转发层，而不是继续充当第二个总控点。

第三步：

- 再立 `PreviewRenderCoordinator`（或等价层），承接 `SelectedImage` 变更后的 preview load 时序，让 `PreviewSelectionState` 继续只负责 UI 输出，不再让它与实际渲染时序纠缠在 `MainWindowViewModel` 里。

第四步：

- 把旧的窗口分支和旧的 preview 刷新出口一起删掉，避免形成“shell coordinator 一轨 + preview render coordinator 一轨 + 旧 MainWindow / ViewModel 逻辑还在”的新三轨。

第五步：

- 完成后必须做 `build + test + 启动 + 测试残留清理`。

### 如果下一任还有余力

- 可以顺手开始把 `PreviewPanel.axaml` 里“文件预览”和“历史/故障/任务分析”视觉上进一步分组，为后续真正拆 `Preview / Task Analysis / History Diagnostics` 三块区域做准备。

## 一个实用判断标准

如果你新增或调整一条规则时，出现了下面任一情况：

- 需要同时改很多个 `ViewModel` 条件分支
- 需要同时补很多个 UI 入口说明
- 需要猜“手动转换”和“监听模式”是否表现一致
- 需要在预警、摘要、提示、历史里重复写相同逻辑

那基本说明这条规则还没有真正进入 `ConversionPlan / WatchProfile / SummaryCoordinator / Policy` 这一层。

这时正确动作通常不是继续堆分支，而是先收口，然后顺手删掉旧出口。
