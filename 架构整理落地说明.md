# 架构整理落地说明

## 文档用途

这份文档不是“重构完成报告”，而是给下一任继续推进时使用的交接说明。

目标只有三个：

- 说明目前已经推进到了哪里
- 说明这一阶段刻意没有动什么
- 说明下一步最应该从哪个中等粒度工作包继续往前推

请继续保持“小步推进、边改边验证、避免一次性大拆”的工作方式，但不要再回到“一次只补一个文案点或一个局部字符串”的推进粒度。

## 一页速览

### 目前已经立起来的边界

- `ApplicationPreferences`：应用级默认偏好
- `ConversionJobDefinition`：当前任务 / 冻结任务的定义
- `WatchProfile`：监听模式配置
- `PreviewToolState` / `PreviewSessionState`：预览工具状态与会话上下文
- `ConversionPlan` / `ConversionRuleSummary` / `ConversionDiagnosticsSummary`：执行前的轻量规划、共享规则摘要、共享诊断摘要
- `RuntimeStatusSummary` / `CompletionSummaryModel`：手动与监听共享的运行中 / 完成后摘要语义基础
- `WatchRuntimeStatusSummary`：监听模式正式的运行中摘要对象，不再只靠局部字符串拼装

### 目前已经落地的关键方向

- 监听模式不再长期直接吃主界面的瞬时状态，而是优先依赖冻结快照
- AI 抠图已经从批量转换 / 预设语义里逐步抽离，回归预览工具语义
- `ConversionPlanningService` 已经开始承担“规则显式化”的职责
- `AI 适用性 / 跳过 AI / 监听模式附加规则` 已经开始共享同一份规则摘要，而不是继续在多个出口各自手拼
- `高压缩 / GIF 转 PDF 大帧 / PDF 锁定跳过 / 体积预估免责声明` 已经开始共享同一层 diagnostics 语义，而不是继续散落在 `ConversionViewModel`
- `ConversionTextPresenter` 已开始承担“完成摘要弹窗 / 历史卡片 / 监听状态文本”的共享格式化职责
- 右侧预览区已经有了“本次任务分析”这类明确的规则出口
- 预览区已经不再直接复用任务级 `CurrentFile`，而是回到“当前选中文件”语义
- 体积预估已经补上 AI / 多页 PDF / 多帧 GIF 的免责声明
- 透明背景被压平成背景填充的规则，已经开始从 `ViewModel` 临时判断收口到 `ConversionPlan`

### 当前仍然故意没有动的高风险区

- 没有直接大拆 `MainWindowViewModel`
- 没有直接大拆 `ImagePreviewWindow`
- 没有全面移除 `AppServices`
- 没有重写整个转换执行主链
- 没有一次性删除旧模型上的全部历史字段

一句话总结当前状态：

> 现在还不是“已经整理完”，而是“已经把正确的推进边界、共享摘要语义和下一段真正该收口的主线打出来了”。

## 到目前为止已经推进了什么

下面只保留真正对后续接手有帮助的主线，不再展开成大段流水账。

### 已经完成的阶段性收口

- 状态边界已经立起来：`ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState`、`PreviewSessionState`、`ConversionPlan` 已经不是概念草图，而是实际开始承接职责。
- 旧设置兼容层已经立起来：`Services/AppSettingsStateMapper.cs` 已经把旧设置对象与新状态模型之间的映射收口，后续迁移不需要再回到“大而全”的旧对象上直接扩字段。
- 监听模式已经从“主界面瞬时状态的附属触发器”往“长期运行任务”推进：启用监听时固化 `ConversionJobDefinition` 快照，监听执行优先吃冻结规则，也支持显式“保存当前参数为监听配置”。
- AI 抠图已经开始回归预览工具语义：相关状态从批量转换 / 预设语义里逐步抽离，避免继续和主批处理管线混用。
- 轻量规划层已经开始接管解释性规则：`ConversionPlanningService` 已承接 AI 是否生效、哪些输入跳过 AI、GIF/PDF 展开、预计输出项、工作项、体积预估免责声明等说明性信息。
- 透明被压平为背景填充的规则，已经不再只是 `ViewModel` 里的局部判断，而是开始收口到 `ConversionPlan`，并被转换前预警、右侧任务分析、当前选中文件告警共同消费。
- 手动转换已经用 `RuntimeStatusSummary` 和 `CompletionSummaryModel` 收拢“运行中状态 / 完成后摘要 / 历史摘要”的共享语义。

### 我这一轮新增推进

- 已新增 `WatchRuntimeState` 与 `WatchRuntimeStatusSummary`，正式给监听模式建立运行中摘要对象，不再继续靠 `WatchStatusText / WatchCurrentFile / IsWatchProcessing` 这一组局部字段各自手拼状态。
- 已扩展 `ConversionStatusSummaryService`，让监听模式也能从同一个摘要服务里拿到 `stopped / waiting / running / processing / completion / validation error / error` 这些状态对象。
- 已新增 `Services/ConversionTextPresenter.cs`，把下面这些原先散在 `MainWindowViewModel` 里的文本格式化开始收口到 presenter：
  - 完成摘要弹窗文本
  - 历史卡片摘要文本
  - 历史卡片详情文本
  - 监听模式状态文本
  - 监听模式统计文本
  - 运行中文件进度文本
- 监听模式处理单个文件时，运行中进度现在会复用 `RuntimeStatusSummary`；监听完成或失败后会落回 `CompletionSummaryModel` 语义，而不是继续走完全旁路的 watch-only 字符串分支。
- `RefreshWatchStatus` 已经改为消费 `WatchRuntimeStatusSummary`，监听模式的 baseline 状态与 terminal 状态不再直接在多个地方重复拼接。
- 完成摘要弹窗、历史卡片、监听状态文本现在已经开始共享同一层 presenter，而不是继续让 `BuildConversionSummaryText`、历史组装、watch 状态文案各自单飞。
- 本轮仍然没有新增 UI 文案键，因此没有扩散 10 语种维护面；后续若再新增文案键，仍要同步简中、繁中、英语、日语、韩语、德语、法语、意大利语、俄语、阿拉伯语这 10 种语言。
- 已补回归测试：`Tests/ImvixPro.Tests/ImageConversionTests.cs` 现在额外覆盖：
  - completion dialog / history detail 的共享 presenter 输出
  - watch runtime summary 的 processing 文本输出
  - watch completion failure 的共享 presenter 输出

### 当前仍未解掉的核心堵点

- 监听模式的运行状态与完成摘要已经不再是明显旁路，但“摘要生命周期协调”仍然散在 `MainWindowViewModel` 与 `MainWindow` 之间，还没有独立的 coordinator。
- `BuildConversionSummaryText` 虽然已经被 presenter 接管了真正格式化逻辑，但完成弹窗触发、历史项落盘、失败日志交接这些动作本身仍然没有一个独立 summary flow 宿主。
- `MainWindowViewModel` 和 `ImagePreviewWindow` 仍然是两个大宿主，边界虽然更清楚了，但结构风险还在。
- `AppServices` 仍然是运行期跨层入口，依赖关系还不够透明。
- `AppSettings` / `ConversionOptions` / `ConversionPreset` 仍然有兼容层噪音，不能再继续横向长字段。

## 下一任最建议从哪里继续推

### 先说结论

下一任不要再回头做“监听状态字符串补丁包”了，这一包已经开始收口。

更适合下一任接手的是一个比本轮再重一点、但仍然可控的工作包：

> **完成摘要协调器与历史出口收口包**

目标不是再补一个提示文本，而是把“manual/watch 完成后发生什么、怎么组装、怎么展示、怎么落盘”这一整类同源逻辑再吃掉一整包。

### 下一任最推荐的中等偏重工作包

建议下一任至少同时完成下面四件事：

1. 给完成摘要建立正式的 coordinator / presenter 入口，不要再让 `MainWindowViewModel` 同时负责生成摘要、写历史、触发弹窗、维护失败日志入口。
2. 把 `BuildHistoryItem`、`BuildConversionSummaryText`、`ConversionCompleted` 这几条链继续往独立 summary flow 收口，让 `MainWindow` 和 `MainWindowViewModel` 都变薄一层。
3. 让 manual/watch 至少在下面三件事上走同一条摘要协调链，而不是只共享底层模型：
   - 完成后摘要对象的生成
   - 历史项的展示模型组装
   - 失败日志入口与展示出口的衔接
4. 做完后顺手删掉 summary flow 周围残留的局部格式化代码和重复出口，避免后续新增规则时又出现“manual 改一份、watch 改一份、window 再改一份”。

### 为什么我建议下一任把范围继续加大

- 监听运行状态这条明显的双轨已经开始被打掉了，再继续回头修 watch status 小点，收益会快速下降。
- 现在真正还在拖慢推进速度的，是“摘要对象已经共享了，但摘要生命周期和展示出口还没有共享”。
- 只要下一任把“summary coordinator + history export/弹窗出口”这一整包拿下，后面再做 `MainWindowViewModel` 拆分、scheduler、`AppServices` 迁移时，阻力会比现在小很多。
- 这个建议仍然没有越线到“大拆主界面”或“重写执行器”，所以风险仍在可控范围内。

### 建议的实际落地顺序

第一步：

- 盘点完成摘要从生成到展示的全路径：`CompletionSummaryModel` 生成、历史项组装、弹窗文本、失败日志路径、watch/manual 触发点。

第二步：

- 先立一个独立 summary coordinator / presenter 入口，再把 `BuildHistoryItem`、`BuildConversionSummaryText` 和相关调用点往外迁。

第三步：

- 让 manual/watch 都走统一的“完成 -> 协调 -> 落盘/弹窗/历史展示”流程，减少摘要出口的重复格式化与重复分支。

第四步：

- 完成后必须做 `build + test + 启动 + 测试残留清理`。

### 如果下一任还有余力

- 可以顺手把 `RecentConversionItem` 的组装也继续往 summary coordinator 下沉。
- 可以开始试着把 `MainWindow.axaml.cs` 里和摘要弹窗强相关的逻辑变薄。
- 但下一轮仍然不建议直接大拆 `MainWindowViewModel`，除非前面的 summary flow 收口已经完成并且验证稳定。

## 一个实用判断标准

如果你新增或调整一条规则时，出现了下面任一情况：

- 需要同时改很多个 `ViewModel` 条件分支
- 需要同时补很多个 UI 入口说明
- 需要猜“手动转换”和“监听模式”是否表现一致
- 需要在预警、摘要、提示、历史里重复写相同逻辑

那基本说明这条规则还没有真正进入 `ConversionPlan / WatchProfile / SummaryCoordinator / Policy` 这一层。

这时正确动作通常不是继续堆分支，而是先收口，然后顺手删掉旧出口。

- 不新增无谓的测试垃圾、临时脚本和一次性验证残留。
- 收尾时清理生成产物，但不要误删真实测试源码和夹具。
- 每次改完至少保证 `build` 通过、主程序可以正常启动。
