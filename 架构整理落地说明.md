# 架构整理落地说明

## 文档用途

这份文档不是“重构完成报告”，而是给下一任继续推进时使用的交接说明。

目标只有三个：

- 说明目前已经推进到了哪里
- 说明这一阶段刻意没有动什么
- 说明下一步最应该从哪个中等粒度工作包继续往前推

请继续保持“小步推进、边改边验证、避免一次性大拆”的工作方式，但不要再回到“一次只补一个文案点或一个局部字符串”的推进粒度。

## 一页速览

### 目前已经立起来的边界

- `ApplicationPreferences`：应用级默认偏好
- `ConversionJobDefinition`：当前任务 / 冻结任务的定义
- `WatchProfile`：监听模式配置
- `PreviewToolState` / `PreviewSessionState`：预览工具状态与会话上下文
- `ConversionPlan` / `ConversionRuleSummary` / `ConversionDiagnosticsSummary`：执行前的轻量规划、共享规则摘要、共享诊断摘要
- `RuntimeStatusSummary` / `CompletionSummaryModel`：手动与监听共享的运行中 / 完成后摘要语义基础
- `WatchRuntimeStatusSummary`：监听模式正式的运行中摘要对象，不再只靠局部字符串拼装
- `ConversionSummaryCoordinator` / `ConversionSummaryFlowResult` / `ConversionSummaryDialogRequest`：手动与监听共享的“完成后摘要 -> 历史档案 -> 弹窗请求”协调层
- `HistoryState`：历史列表、历史持久化读取、历史卡片展示文本、清空动作的独立子域
- `NotificationState`：失败日志入口、待展示完成弹窗请求的独立子域

### 目前已经落地的关键方向

- 监听模式不再长期直接吃主界面的瞬时状态，而是优先依赖冻结快照
- AI 抠图已经从批量转换 / 预设语义里逐步抽离，回归预览工具语义
- `ConversionPlanningService` 已经开始承担“规则显式化”的职责
- `AI 适用性 / 跳过 AI / 监听模式附加规则` 已经开始共享同一份规则摘要，而不是继续在多个出口各自手拼
- `高压缩 / GIF 转 PDF 大帧 / PDF 锁定跳过 / 体积预估免责声明` 已经开始共享同一层 diagnostics 语义，而不是继续散落在 `ConversionViewModel`
- `ConversionTextPresenter` 已开始承担“完成摘要弹窗 / 历史卡片 / 监听状态文本”的共享格式化职责
- `ConversionSummaryCoordinator` 已开始承担“完成后摘要生命周期协调”的职责，manual/watch 不再各自手拼历史项与弹窗文案
- `HistoryState` / `NotificationState` 已经把“完成后遗留状态”从 `MainWindowViewModel` 主体里拆出，主 ViewModel 不再直接持有历史集合与失败日志路径
- `MainWindow.axaml.cs` 不再订阅手工 completion event，而是改为消费 `NotificationState` 上的待展示弹窗请求
- 右侧预览区已经有了“本次任务分析”这类明确的规则出口
- 预览区已经不再直接复用任务级 `CurrentFile`，而是回到“当前选中文件”语义
- 体积预估已经补上 AI / 多页 PDF / 多帧 GIF 的免责声明
- 透明背景被压平成背景填充的规则，已经开始从 `ViewModel` 临时判断收口到 `ConversionPlan`

### 当前仍然故意没有动的高风险区

- 没有直接大拆 `MainWindowViewModel`
- 没有直接大拆 `ImagePreviewWindow`
- 没有全面移除 `AppServices`
- 没有重写整个转换执行主链
- 没有一次性删除旧模型上的全部历史字段

一句话总结当前状态：

> 现在还不是“已经整理完”，而是“已经把正确的推进边界、共享摘要语义和下一段真正该收口的主线打出来了”。

## 到目前为止已经推进了什么

下面只保留真正对后续接手有帮助的主线，不再展开成大段流水账。

### 已经完成的阶段性收口

- 状态边界已经立起来：`ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState`、`PreviewSessionState`、`ConversionPlan` 已经不是概念草图，而是实际开始承接职责。
- 旧设置兼容层已经立起来：`Services/AppSettingsStateMapper.cs` 已经把旧设置对象与新状态模型之间的映射收口，后续迁移不需要再回到“大而全”的旧对象上直接扩字段。
- 监听模式已经从“主界面瞬时状态的附属触发器”往“长期运行任务”推进：启用监听时固化 `ConversionJobDefinition` 快照，监听执行优先吃冻结规则，也支持显式“保存当前参数为监听配置”。
- AI 抠图已经开始回归预览工具语义：相关状态从批量转换 / 预设语义里逐步抽离，避免继续和主批处理管线混用。
- 轻量规划层已经开始接管解释性规则：`ConversionPlanningService` 已承接 AI 是否生效、哪些输入跳过 AI、GIF/PDF 展开、预计输出项、工作项、体积预估免责声明等说明性信息。
- 透明被压平为背景填充的规则，已经不再只是 `ViewModel` 里的局部判断，而是开始收口到 `ConversionPlan`，并被转换前预警、右侧任务分析、当前选中文件告警共同消费。
- 手动转换已经用 `RuntimeStatusSummary` 和 `CompletionSummaryModel` 收拢“运行中状态 / 完成后摘要 / 历史摘要”的共享语义。

### 我这一轮新增推进

- 已新增 `ViewModels/Main/HistoryState.cs`，把历史缓存、历史加载、历史持久化追加、历史卡片文本重建、清空历史动作收口成独立子域，`MainWindowViewModel` 不再直接持有 `_historyCache` 与 `RecentConversions`。
- 已新增 `ViewModels/Main/NotificationState.cs`，把失败日志入口与待展示完成弹窗请求独立出来，`MainWindowViewModel` 不再直接持有 `LastFailureLogPath` 这类完成后残留状态。
- 手动转换完成后现在会先把 `ConversionSummaryFlowResult` 交给 `NotificationState` 与 `HistoryState`，再由 UI 消费这些结果态，而不是继续让主 ViewModel 自己维护历史集合与完成通知后续状态。
- 监听模式完成后也已经改成直接喂给 `HistoryState`，manual/watch 两条线现在共享同一个历史状态宿主，而不是再分别维护历史出口。
- `Views/MainWindow.axaml.cs` 这一轮继续变薄：窗口层不再依赖 `ConversionCompleted` 事件，而是只订阅 `NotificationState.PendingDialogRequest` 并在窗口可见时消费它；这让“待展示弹窗请求”终于变成了状态，而不是瞬时事件。
- `Views/Controls/PreviewPanel.axaml` 已改为直接绑定 `HistoryState` / `NotificationState`，右侧面板里的历史与失败日志区域不再通过主 ViewModel 的扁平属性转发。
- 本轮顺手删掉了围绕 history / notification 的旧扁平出口，避免形成“新 state 已经有了，但旧字段还在继续同步”的双轨。
- 本轮仍然没有新增 UI 文案键，因此没有扩散 10 语种维护面；后续若再新增文案键，仍要同步简中、繁中、英语、日语、韩语、德语、法语、意大利语、俄语、阿拉伯语这 10 种语言。
- 已补回归测试：`Tests/ImvixPro.Tests/ImageConversionTests.cs` 现在额外覆盖 `NotificationState` 的 pending dialog 生命周期与 failure log 重置行为，确保新的通知子域不是纯结构搬家而是可验证的状态收口。

### 当前仍未解掉的核心堵点

- history / notification 虽然已经独立成 `HistoryState` / `NotificationState`，但“任务分析 + 风险提示 + 体积预估 + 格式推荐”这整块右侧面板状态仍然直接挂在 `MainWindowViewModel` 上，`RefreshConversionInsights` 仍然是一个大收口点。
- `MainWindow.axaml.cs` 已经从 completion event 退到消费 `NotificationState`，但文件选择、预览打开、Tray、窗口宿主逻辑仍然集中在同一个窗口代码后置文件里，没有进一步形成更薄的 shell 协调层。
- `MainWindowViewModel` 和 `ImagePreviewWindow` 仍然是两个大宿主，边界虽然更清楚了，但结构风险还在。
- `AppServices` 仍然是运行期跨层入口，依赖关系还不够透明。
- `AppSettings` / `ConversionOptions` / `ConversionPreset` 仍然有兼容层噪音，不能再继续横向长字段。

## 下一任最建议从哪里继续推

### 先说结论

下一任不要再回头做 `history + notification` 这一包的补丁了，这一包这轮已经正式收口了，继续在这里小修只会重新掉回“边角优化”。

更适合下一任接手的是一个比本轮更重、而且应该明确做成“子域抽离”的工作包：

> **PreviewSidebarState / TaskAnalysisState 收口包**

目标不再是继续收完成后状态，而是把右侧面板里仍然挂在 `MainWindowViewModel` 上的“任务分析 + 风险提示 + 体积预估 + 格式推荐”这整片子域一起拿下来，直接减少主 ViewModel 的另一整片负担。

### 下一任最推荐的中等偏重工作包

建议下一任至少同时完成下面五件事：

1. 把 `ActiveWarnings`、`ConversionPlanHighlights`、`FormatRecommendationText`、`FormatRecommendationReasonText`、`OriginalSizeSummaryText`、`EstimatedSizeSummaryText`、`EstimateDisclaimerText` 这一整组字段从 `MainWindowViewModel` 收口到独立 `PreviewSidebarState` 或 `TaskAnalysisState`，不要再让主 ViewModel 继续平铺维护右侧分析面板的全部展示状态。
2. 把 `RefreshConversionInsights()` 从“同时直接改很多个主 ViewModel 属性”的大方法，改造成“生成一份分析结果 -> 交给独立 state 更新”的收口链，顺手把 `RefreshActiveWarnings()` / `RefreshConversionPlanInsights()` 这类细碎更新点并到同一条子域刷新路径里。
3. 让 `PreviewPanel.axaml` 的“本次任务分析 / 风险提示 / 格式推荐 / 体积预估 / 历史与故障”绑定更明显地区分“当前选中文件语义”和“任务级分析语义”，不要再让两类信息继续在主 ViewModel 上混成一片。
4. 做这一包时顺手把 `NotificationState`、`HistoryState`、新的 `TaskAnalysisState` 在主 ViewModel 里排成同一层 feature state，而不是继续新增零散扁平属性；也就是说，这一轮要开始明确 `MainWindowViewModel` 只是壳，不是所有状态的最终宿主。
5. 做完后必须把旧的扁平分析字段和旧刷新出口一并删掉，不要留下“新 sidebar state 已经有了，但 `MainWindowViewModel` 里原字段还在同步”的双轨。

### 为什么我建议下一任把范围继续加大

- 这轮已经把“完成后遗留状态”收掉了，再回头继续抠这一块，收益会明显下降。

- 这个建议虽然比上一轮更重，但仍然没有越线到“重写执行主链”或“直接大拆主界面”，风险仍是可控的，而且比继续做小补丁更接近真正提速。

## 一个实用判断标准

如果你新增或调整一条规则时，出现了下面任一情况：

- 需要同时改很多个 `ViewModel` 条件分支
- 需要同时补很多个 UI 入口说明
- 需要猜“手动转换”和“监听模式”是否表现一致
- 需要在预警、摘要、提示、历史里重复写相同逻辑

那基本说明这条规则还没有真正进入 `ConversionPlan / WatchProfile / SummaryCoordinator / Policy` 这一层。

这时正确动作通常不是继续堆分支，而是先收口，然后顺手删掉旧出口。
