# 架构整理落地说明

## 文档用途

这份文档不是“重构完成报告”，而是给下一任继续推进时使用的交接说明。

目标只有三个：

- 说明目前已经推进到了哪里
- 说明这一阶段刻意没有动什么
- 说明下一步最应该从哪个中等粒度工作包继续往前推

请继续保持“小步推进、边改边验证、避免一次性大拆”的工作方式，但不要再回到“一次只补一个文案点或一个局部字符串”的推进粒度。

## 一页速览

### 目前已经立起来的边界

- `ApplicationPreferences`：应用级默认偏好
- `ConversionJobDefinition`：当前任务 / 冻结任务的定义
- `WatchProfile`：监听模式配置
- `PreviewToolState` / `PreviewSessionState`：预览工具状态与会话上下文
- `ConversionPlan` / `ConversionRuleSummary` / `ConversionDiagnosticsSummary`：执行前的轻量规划、共享规则摘要、共享诊断摘要
- `RuntimeStatusSummary` / `CompletionSummaryModel`：手动与监听共享的运行中 / 完成后摘要语义基础
- `WatchRuntimeStatusSummary`：监听模式正式的运行中摘要对象，不再只靠局部字符串拼装
- `ConversionSummaryCoordinator` / `ConversionSummaryFlowResult` / `ConversionSummaryDialogRequest`：手动与监听共享的“完成后摘要 -> 历史档案 -> 弹窗请求”协调层

### 目前已经落地的关键方向

- 监听模式不再长期直接吃主界面的瞬时状态，而是优先依赖冻结快照
- AI 抠图已经从批量转换 / 预设语义里逐步抽离，回归预览工具语义
- `ConversionPlanningService` 已经开始承担“规则显式化”的职责
- `AI 适用性 / 跳过 AI / 监听模式附加规则` 已经开始共享同一份规则摘要，而不是继续在多个出口各自手拼
- `高压缩 / GIF 转 PDF 大帧 / PDF 锁定跳过 / 体积预估免责声明` 已经开始共享同一层 diagnostics 语义，而不是继续散落在 `ConversionViewModel`
- `ConversionTextPresenter` 已开始承担“完成摘要弹窗 / 历史卡片 / 监听状态文本”的共享格式化职责
- `ConversionSummaryCoordinator` 已开始承担“完成后摘要生命周期协调”的职责，manual/watch 不再各自手拼历史项与弹窗文案
- 右侧预览区已经有了“本次任务分析”这类明确的规则出口
- 预览区已经不再直接复用任务级 `CurrentFile`，而是回到“当前选中文件”语义
- 体积预估已经补上 AI / 多页 PDF / 多帧 GIF 的免责声明
- 透明背景被压平成背景填充的规则，已经开始从 `ViewModel` 临时判断收口到 `ConversionPlan`

### 当前仍然故意没有动的高风险区

- 没有直接大拆 `MainWindowViewModel`
- 没有直接大拆 `ImagePreviewWindow`
- 没有全面移除 `AppServices`
- 没有重写整个转换执行主链
- 没有一次性删除旧模型上的全部历史字段

一句话总结当前状态：

> 现在还不是“已经整理完”，而是“已经把正确的推进边界、共享摘要语义和下一段真正该收口的主线打出来了”。

## 到目前为止已经推进了什么

下面只保留真正对后续接手有帮助的主线，不再展开成大段流水账。

### 已经完成的阶段性收口

- 状态边界已经立起来：`ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState`、`PreviewSessionState`、`ConversionPlan` 已经不是概念草图，而是实际开始承接职责。
- 旧设置兼容层已经立起来：`Services/AppSettingsStateMapper.cs` 已经把旧设置对象与新状态模型之间的映射收口，后续迁移不需要再回到“大而全”的旧对象上直接扩字段。
- 监听模式已经从“主界面瞬时状态的附属触发器”往“长期运行任务”推进：启用监听时固化 `ConversionJobDefinition` 快照，监听执行优先吃冻结规则，也支持显式“保存当前参数为监听配置”。
- AI 抠图已经开始回归预览工具语义：相关状态从批量转换 / 预设语义里逐步抽离，避免继续和主批处理管线混用。
- 轻量规划层已经开始接管解释性规则：`ConversionPlanningService` 已承接 AI 是否生效、哪些输入跳过 AI、GIF/PDF 展开、预计输出项、工作项、体积预估免责声明等说明性信息。
- 透明被压平为背景填充的规则，已经不再只是 `ViewModel` 里的局部判断，而是开始收口到 `ConversionPlan`，并被转换前预警、右侧任务分析、当前选中文件告警共同消费。
- 手动转换已经用 `RuntimeStatusSummary` 和 `CompletionSummaryModel` 收拢“运行中状态 / 完成后摘要 / 历史摘要”的共享语义。

### 我这一轮新增推进

- 已新增 `Services/ConversionSummaryCoordinator.cs`，正式把“完成后摘要对象生成、历史档案生成、历史展示项生成、弹窗请求生成”收口到同一层协调器，而不是继续分散在 `MainWindowViewModel`、`MainWindow`、历史组装方法里。
- 已新增 `ConversionSummaryFlowResult` 与 `ConversionSummaryDialogRequest`，让“完成后应该展示什么”从原来的裸 `CompletionSummaryModel` 变成结构化结果，避免窗口层再自己拼标题、正文、按钮文案。
- 手动转换现在完成后会统一走 `BuildCompletionFlow(..., includeDialog: true)`：
  - 生成共享 `CompletionSummaryModel`
  - 生成历史档案 `ConversionHistoryEntry`
  - 生成弹窗请求 `ConversionSummaryDialogRequest`
  - 然后再分别驱动状态栏、历史、完成弹窗
- 监听模式现在完成后也统一走 `BuildCompletionFlow(..., includeDialog: false)`，与手动模式共用同一条“完成摘要 -> 历史档案”链，只在是否弹窗这一层做分流。
- `ReplaceHistory` 不再自己调用 `BuildHistoryItem` 手工拼历史卡片，而是统一改为走 coordinator 的 `CreateHistoryItem`，这样历史摘要文本、历史详情文本与完成摘要文本都回到同一个 summary flow 语义源头。
- `Views/MainWindow.axaml.cs` 已变薄：窗口层不再调用 `BuildConversionSummaryText` 自己格式化摘要正文，而是只消费 `ConversionSummaryDialogRequest` 并负责展示。
- 本轮已删掉 `BuildHistoryItem` 与 `BuildConversionSummaryText` 这类局部出口，减少后续再出现“改完 presenter 还要改 window / history / manual 分支”的回弹点。
- 本轮仍然没有新增 UI 文案键，因此没有扩散 10 语种维护面；后续若再新增文案键，仍要同步简中、繁中、英语、日语、韩语、德语、法语、意大利语、俄语、阿拉伯语这 10 种语言。
- 已补回归测试：`Tests/ImvixPro.Tests/ImageConversionTests.cs` 现在额外覆盖：
  - `ConversionSummaryCoordinator` 的 manual completion dialog/history 统一输出
  - `ConversionSummaryCoordinator` 的 watch completion 无弹窗但保留历史档案
  - 既有 shared presenter 的 dialog/history/watch 文本输出仍保持稳定

### 当前仍未解掉的核心堵点

- manual/watch 的完成后摘要已经开始走统一 coordinator，但“历史状态本身”和“窗口级通知宿主”仍然挂在 `MainWindowViewModel` / `MainWindow` 上，没有真正形成独立子域。
- `BuildConversionSummaryText` 与 `BuildHistoryItem` 这两类局部出口已经被收掉，但 `_historyCache`、`RecentConversions`、`LastFailureLogPath` 这些摘要后续状态仍然没有单独的 history/notification state 宿主。
- `MainWindowViewModel` 和 `ImagePreviewWindow` 仍然是两个大宿主，边界虽然更清楚了，但结构风险还在。
- `AppServices` 仍然是运行期跨层入口，依赖关系还不够透明。
- `AppSettings` / `ConversionOptions` / `ConversionPreset` 仍然有兼容层噪音，不能再继续横向长字段。

## 下一任最建议从哪里继续推

### 先说结论

下一任不要再回头做“完成摘要弹窗 / 历史卡片文案小补丁”了，这一包这轮已经正式收口了一大半。

更适合下一任接手的是一个比本轮更重、而且应该明确做成“子域抽离”的工作包：

> **HistoryState + NotificationState 下沉包**

目标不再是补 presenter，而是把“完成后留下的状态”也一起吃掉：历史列表、失败日志入口、待展示弹窗请求、清空历史动作，不要再继续堆在 `MainWindowViewModel` 这个大宿主上。

### 下一任最推荐的中等偏重工作包

建议下一任至少同时完成下面五件事：

1. 把 `_historyCache`、`RecentConversions`、`ClearRecentConversions`、`LoadRecentConversionHistory` 这整组职责从 `MainWindowViewModel` 里继续往独立 `HistoryState` / history coordinator 下沉，不要再让主 ViewModel 同时承担“任务状态 + 历史状态 + 历史持久化”三层职责。
2. 把 `LastFailureLogPath` 和完成后的通知生命周期一起收口成独立 notification state，让“手动完成后弹窗 / 失败日志入口 / 最后一次失败信息”不再散落为主 ViewModel 上的局部属性。
3. 让 `MainWindow.axaml.cs` 不只是“消费 dialog request”，而是进一步减少对完成摘要流的直接耦合，最好退化为只负责显示 notification state，而不是继续扮演 summary flow 宿主。
4. 顺手把 `RecentConversionItem` 的组装和历史加载后的 UI 刷新链也收掉，避免后面做 `MainWindowViewModel` 真拆分时又要回头处理历史列表刷新逻辑。
5. 做完后必须把 summary/history/notification 周围残留的重复入口一并删掉，不要留“新 coordinator 已经有了，但主 ViewModel 上的旧出口还在”的双轨状态。

### 为什么我建议下一任把范围继续加大

- 这轮已经把“完成后摘要对象、历史档案、弹窗请求”统一了，再继续做局部 presenter 调整，收益会明显下降。
- 现在真正拖慢推进速度的，已经从“摘要怎么生成”变成“摘要生成以后，哪些状态留在谁那里”。
- 只要下一任把 `history + notification` 这一整片子域一起下沉，后面再做 `MainWindowViewModel` 拆分、`AppServices` 迁移、scheduler 收口时，都会轻很多。
- 这个建议虽然比上一轮更重，但仍然没有越线到“重写执行主链”或“直接大拆主界面”，风险仍是可控的。

### 建议的实际落地顺序

第一步：

- 盘点 `MainWindowViewModel` 里与“完成后遗留状态”相关的全部内容：`_historyCache`、`RecentConversions`、`LastFailureLogPath`、`LoadRecentConversionHistory`、`AppendHistory`、`ClearRecentConversions`、窗口弹窗请求出口。

第二步：

- 先立一个独立 `HistoryState` / notification coordinator 入口，让手动与监听完成后都只把 `ConversionSummaryFlowResult` 交给它，而不是继续自己维护后续状态。

第三步：

- 把 `RecentConversions` 的组装、历史持久化后的替换、失败日志入口刷新这三条链一起迁过去，并让 `MainWindow.axaml.cs` 只消费一个更薄的窗口通知接口。

第四步：

- 做完后顺手删掉 `MainWindowViewModel` 里围绕 history/notification 残留的旧出口与旧字段，避免形成新的双轨。

第五步：

- 完成后必须做 `build + test + 启动 + 测试残留清理`。

### 如果下一任还有余力

- 可以顺手把 history/notification 这一块开始抽成真正独立的 feature state，而不只是 coordinator。
- 可以开始试着把 `MainWindow.axaml.cs` 里与完成通知强相关的逻辑进一步压到最薄。
- 但下一轮仍然不建议直接大拆 `MainWindowViewModel` 的全部 partial，除非前面的 history/notification 子域已经收稳并且验证稳定。

## 一个实用判断标准

如果你新增或调整一条规则时，出现了下面任一情况：

- 需要同时改很多个 `ViewModel` 条件分支
- 需要同时补很多个 UI 入口说明
- 需要猜“手动转换”和“监听模式”是否表现一致
- 需要在预警、摘要、提示、历史里重复写相同逻辑

那基本说明这条规则还没有真正进入 `ConversionPlan / WatchProfile / SummaryCoordinator / Policy` 这一层。

这时正确动作通常不是继续堆分支，而是先收口，然后顺手删掉旧出口。

## 建议继续保持的工作方式

- 不要再按“一个极小点一轮”推进，改成“一个中等范围规则包一轮”。
- 先做“边界更清楚、规则出口更统一”，再做“大结构更漂亮”。
- 不新增无谓的测试垃圾、临时脚本和一次性验证残留。
- 收尾时清理生成产物，但不要误删真实测试源码和夹具。
- 每次改完至少保证 `build` 通过、主程序可以正常启动。
