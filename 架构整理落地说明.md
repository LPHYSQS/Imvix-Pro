# 架构整理落地说明

## 文档用途

这份文档不是“重构完成报告”，而是给下一任继续推进时使用的交接说明。

目标只有三个：

- 说明目前已经推进到了哪里
- 说明这一阶段刻意没有动什么
- 说明下一步最应该从哪个小切口继续往前推

请继续保持“小步推进、边改边验证、避免一次性大拆”的工作方式。

---

## 一页速览

### 目前已经立起来的边界

- `ApplicationPreferences`：应用级默认偏好
- `ConversionJobDefinition`：当前任务 / 冻结任务的定义
- `WatchProfile`：监听模式配置
- `PreviewToolState` / `PreviewSessionState`：预览工具状态与会话上下文
- `ConversionPlan` / `ConversionRuleSummary` / `ConversionDiagnosticsSummary`：执行前的轻量规划、共享规则摘要、共享诊断摘要

### 目前已经落地的关键方向

- 监听模式不再长期直接吃主界面的瞬时状态，而是优先依赖冻结快照
- AI 抠图已经从批量转换 / 预设语义里逐步抽离，回归预览工具语义
- `ConversionPlanningService` 已经开始承担“规则显式化”的职责
- `AI 适用性 / 跳过 AI / 监听模式附加规则` 已经开始共享同一份规则摘要，而不是继续在多个出口各自手拼
- `高压缩 / GIF 转 PDF 大帧 / PDF 锁定跳过 / 体积预估免责声明` 已经开始共享同一层 diagnostics 语义，而不是继续散落在 `ConversionViewModel`
- 右侧预览区已经有了“本次任务分析”这类明确的规则出口
- 体积预估已经补上 AI / 多页 PDF / 多帧 GIF 的免责声明
- 透明背景被压平成背景填充的规则，已经开始从 `ViewModel` 临时判断收口到 `ConversionPlan`

### 当前仍然故意没有动的高风险区

- 没有直接大拆 `MainWindowViewModel`
- 没有直接大拆 `ImagePreviewWindow`
- 没有全面移除 `AppServices`
- 没有重写整个转换执行主链
- 没有一次性删除旧模型上的全部历史字段

一句话总结当前状态：

> 现在还不是“已经整理完”，而是“已经把正确的推进边界和主方向打出来了”。

## 到目前为止已经推进了什么

下面改成压缩归纳版，只保留真正对后续接手有帮助的主线。

### 已经完成的阶段性收口

- 状态边界已经立起来：`ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState`、`PreviewSessionState`、`ConversionPlan` 已经不是概念草图，而是实际开始承接职责。
- 旧设置兼容层已经立起来：`Services/AppSettingsStateMapper.cs` 已经把旧设置对象与新状态模型之间的映射收口，后续迁移不需要再回到“大而全”的旧对象上直接扩字段。
- 监听模式已经从“主界面瞬时状态的附属触发器”往“长期运行任务”推进：启用监听时固化 `ConversionJobDefinition` 快照，监听执行优先吃冻结规则，也支持显式“保存当前参数为监听配置”。
- AI 抠图已经开始回归预览工具语义：相关状态从批量转换 / 预设语义里逐步抽离，避免继续和主批处理管线混用。
- 轻量规划层已经开始接管解释性规则：`ConversionPlanningService` 已承接 AI 是否生效、哪些输入跳过 AI、GIF/PDF 展开、预计输出项、工作项、体积预估免责声明等说明性信息。
- 透明被压平为背景填充的规则，已经不再只是 `ViewModel` 里的局部判断，而是开始收口到 `ConversionPlan`，并被转换前预警、右侧任务分析、当前选中文件告警共同消费。

### 我这一轮新增推进

- 新增了共享诊断摘要对象：`ConversionDiagnosticsSummary`，把高压缩风险、GIF 转 PDF 大帧风险、PDF 锁定跳过、体积预估免责声明这几类仍留在 `ViewModel` 里的诊断语义，继续往规划层收口。
- `ConversionPlanningService` 现在产出的 `ConversionPlan` 已同时带上 `RuleSummary + DiagnosticsSummary`，不再只有规则摘要，没有诊断摘要。
- 手动转换前确认弹窗、右侧 warnings 区、`ActiveWarnings` 这三处现在已经开始消费同一份 diagnostics 结果，不再分别手拼高压缩 / GIF-PDF / PDF 锁定等条件分支。
- `ConversionViewModel` 里原本重复的 `IsHighCompressionSelection`、`GetLargeGifPdfFrameCount`、局部免责声明组装已经收掉一批，重复 if/字符串拼装明显减少。
- 本轮没有新增 UI 文案键，因此没有再扩散 10 语种维护面；这也是刻意控制变更面的结果。后续若再新增文案键，仍要同步简中、繁中、英语、日语、韩语、德语、法语、意大利语、俄语、阿拉伯语这 10 种语言。
- 已补轻量回归测试：`Tests/ImvixPro.Tests/ImageConversionTests.cs` 现在覆盖了 diagnostics 摘要对高压缩、AI 免责声明、GIF 转 PDF 大帧、PDF 锁定跳过的稳定表达。

### 当前仍未解掉的核心堵点

- 现在规则摘要和诊断摘要都已经开始收口，但“运行中状态 / 一次性完成摘要 / 历史卡片摘要”这三层信息仍没有统一模型。
- `CurrentFile`、底部状态栏、完成后摘要、历史卡片之间，仍在复用和重复表达同一批任务信息，语义边界还不够清晰。
- 监听模式虽然已经有冻结快照，但“监听执行中的状态摘要”和“手动执行完成后的摘要”还没有共享统一的 completion/runtime 语义。
- `MainWindowViewModel` 和 `ImagePreviewWindow` 仍然是两个大宿主，边界虽然开始清楚，但结构风险还在。
- `AppServices` 仍然是运行期跨层入口，依赖关系还不够透明。
- `AppSettings` / `ConversionOptions` / `ConversionPreset` 仍然有兼容层噪音，不能再继续横向长字段。

为了控制风险，下面这些事情仍然是有意识地先不硬拆：

- 不直接大拆 `MainWindowViewModel` partial 体系。
- 不直接大拆 `ImagePreviewWindow` 工具宿主。

- 不直接把 `AppServices` 一次性拿掉。

原因不变：

> 如果在规则层还没进一步收稳的时候，同时叠加结构拆分、依赖迁移、执行器重写，风险会急剧上升。

## 下一任最建议从哪里继续推

### 先说结论

下一任不要再回到“一个文案点 / 一个局部方法 / 一个提示出口”这种过小切口了，也不要重复再做一轮 diagnostics 收口的尾巴活。

更合适的粒度应该是：

- 一次拿下一整类同源规则。
- 一次至少收掉这类规则在 2 到 3 个出口上的重复实现。
- 一次做完之后，要能真实减少后续轮次，而不是只让文档里的“还要 6 到 8 次”保持不变。

### 下一任最推荐的中等偏重工作包

建议下一任直接做一个“状态与完成摘要统一包”，范围要比这一轮再重一点，但仍然控制在“中等偏重、非失控式重构”的范围内。

这一包建议至少包含下面四件事：

1. 新增 `RuntimeStatusSummary` / `CompletionSummaryModel` / 等价对象，把下面这组当前还在不同出口各自组织的语义，收成同一层可复用结果：
- 实时状态
- 当前文件进度
- 完成后摘要
- 失败计数 / 取消状态
- Failure log 入口摘要
2. 让下面至少三个出口尽量消费同一份 completion/runtime 结果，而不是继续各自讲一遍：
- 底部状态栏
- 完成后摘要窗口或等价出口
- 历史卡片摘要
3. 明确把“当前正在处理的文件名”和“本次任务完成后的摘要文本”拆开，不要再让 `CurrentFile` 一类字段兼任两种语义。
4. 如果时间还够，把监听模式的运行中摘要也一起接到同一套 completion/runtime 语义上，避免监听模式继续走旁路文案。

### 为什么我建议下一任把范围加大

- diagnostics 这一包已经收进去一块，现在真正卡进度的，已经是“同一任务信息在多个状态/摘要出口重复表达”的问题。
- 如果继续按“一轮只修一个很小点”的方式推进，剩余轮次不会明显下降。
- 反过来，只要下一任一次吃掉“状态与完成摘要”这一整包，后面再处理 ViewModel 拆分、scheduler、AppServices 时，入口就会清楚得多。
- 这一建议仍然没有越线到“大拆主界面”或“重写执行器”，所以风险仍在可控范围内。

### 建议的实际落地顺序

- 第一步：盘点底部状态栏、完成后摘要、历史卡片、监听状态这几处，到底哪些字段在重复讲同一件事。
- 第二步：先在模型层立起统一的 runtime/completion summary 对象，不要一上来改界面文案。
- 第三步：优先替换底部状态栏、完成摘要、历史卡片三处出口，再删旧字段复用和旧分支。
- 第四步：完成后必须做 `build + 启动 + 生成残留清理`。

### 如果下一任还有余力

- 可以顺手继续盘点 `CurrentFile`、状态栏文案、历史摘要里哪些字段其实应该拆成 `RuntimeStatus` 与 `CompletionSummary` 两类。
- 但下一轮仍然不建议直接大拆 `MainWindowViewModel`，除非前面的状态/摘要出口统一已经完成并且验证稳定。

## 下一任暂时不建议马上做的事

- 不建议直接全面拆 `MainWindowViewModel`
- 不建议直接全面迁移 `AppServices`
- 不建议一边重写执行主链、一边重做预览窗口、一边收规则层
- 不建议为了表面整洁一次性删掉大量旧字段

当前更好的策略是：

> 先用一个中等范围工作包，把同源规则的多个出口一起收掉，再考虑更大的结构动作。

## 一个实用判断标准

如果你新增或调整一条规则时，出现了下面任一情况：

- 需要同时改很多个 `ViewModel` 条件分支
- 需要同时补很多个 UI 入口说明
- 需要猜“手动转换”和“监听模式”是否表现一致
- 需要在预警、摘要、提示里重复写相同逻辑

那基本说明这条规则还没有真正进入 `ConversionPlan / WatchProfile / Policy` 这一层。

这时正确动作通常不是继续堆分支，而是先收口，然后顺手删掉旧出口。

## 建议继续保持的工作方式

- 不要再按“一个极小点一轮”推进，改成“一个中等范围规则包一轮”。
- 先做“边界更清楚、规则出口更统一”，再做“大结构更漂亮”。
- 不新增无谓的测试垃圾、临时脚本和一次性验证残留。
- 收尾时清理生成产物，但不要误删真实测试源码和夹具。
- 每次改完至少保证 `build` 通过、主程序可以正常启动。

## 给下一任的最后一句话

这份工作到现在，已经不适合再按“一个提醒文案一轮”或“一个 diagnostics 小尾巴一轮”去推进了。

请至少拿下一个“中等范围状态/摘要包”，让剩余轮次真的开始下降。

判断你这轮有没有做对，不是看你加了多少新类，而是看你有没有同时做到下面三件事：

- 多统一一整类状态/摘要出口
- 少留一处重复判断和双轨逻辑
- 改完还能稳定 `build + 启动`

只要做到这一点，这个项目就会开始明显从“渐进止血”转向“可持续维护”。
