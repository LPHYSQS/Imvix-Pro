# 架构整理落地说明

## 文档用途

这份文档不是“重构完成报告”，而是给下一任继续推进时使用的交接说明。

目标只有三个：

- 说明目前已经推进到了哪里
- 说明这一阶段刻意没有动什么
- 说明下一步最应该从哪个中等粒度工作包继续往前推

请继续保持“小步推进、边改边验证、避免一次性大拆”的工作方式，但不要再回到“一次只补一个文案点或一个局部字符串”的推进粒度。

## 一页速览

### 目前已经立起来的边界

- `ApplicationPreferences`：应用级默认偏好
- `ConversionJobDefinition`：当前任务 / 冻结任务的定义
- `WatchProfile`：监听模式配置
- `PreviewToolState` / `PreviewSessionState`：预览工具状态与会话上下文
- `PreviewSelectionSnapshot` / `PreviewSelectionState`：当前选中文件预览、PDF/GIF/PSD 文件级可见性与预览面板显示状态的独立子域
- `ConversionPlan` / `ConversionRuleSummary` / `ConversionDiagnosticsSummary`：执行前的轻量规划、共享规则摘要、共享诊断摘要
- `RuntimeStatusSummary` / `CompletionSummaryModel`：手动与监听共享的运行中 / 完成后摘要语义基础
- `WatchRuntimeStatusSummary`：监听模式正式的运行中摘要对象，不再只靠局部字符串拼装
- `ConversionSummaryCoordinator` / `ConversionSummaryFlowResult` / `ConversionSummaryDialogRequest`：手动与监听共享的“完成后摘要 -> 历史档案 -> 弹窗请求”协调层
- `HistoryState`：历史列表、历史持久化读取、历史卡片展示文本、清空动作的独立子域
- `NotificationState`：失败日志入口、待展示完成弹窗请求的独立子域
- `TaskAnalysisState`：右侧任务分析面板的警告、规则摘要、格式推荐、体积预估与免责声明子域

### 目前已经落地的关键方向

- 监听模式不再长期直接吃主界面的瞬时状态，而是优先依赖冻结快照
- AI 抠图已经从批量转换 / 预设语义里逐步抽离，回归预览工具语义
- `ConversionPlanningService` 已经开始承担“规则显式化”的职责
- `AI 适用性 / 跳过 AI / 监听模式附加规则` 已经开始共享同一份规则摘要，而不是继续在多个出口各自手拼
- `高压缩 / GIF 转 PDF 大帧 / PDF 锁定跳过 / 体积预估免责声明` 已经开始共享同一层 diagnostics 语义，而不是继续散落在 `ConversionViewModel`
- `ConversionTextPresenter` 已开始承担“完成摘要弹窗 / 历史卡片 / 监听状态文本”的共享格式化职责
- `ConversionSummaryCoordinator` 已开始承担“完成后摘要生命周期协调”的职责，manual/watch 不再各自手拼历史项与弹窗文案
- `HistoryState` / `NotificationState` 已经把“完成后遗留状态”从 `MainWindowViewModel` 主体里拆出，主 ViewModel 不再直接持有历史集合与失败日志路径
- `TaskAnalysisState` 已经把“任务分析 + 风险提示 + 格式推荐 + 体积预估”从 `MainWindowViewModel` 的扁平字段里拆出，`RefreshConversionInsights()` 现在改为“生成分析快照 -> 更新子域状态”
- `MainWindow.axaml.cs` 不再订阅手工 completion event，而是改为消费 `NotificationState` 上的待展示弹窗请求
- `MainWindowShellCoordinator` 已经接手 completion dialog 消费、第二实例提示、Tray hide/restore/exit、BringToFront，以及预览窗口 / 文件详情窗口 / PDF 解锁弹窗的宿主协调入口
- 右侧预览区已经有了“本次任务分析”这类明确的规则出口
- `PreviewPanel.axaml` 的警告 / 任务分析 / 格式推荐 / 体积预估四块现在已经直接绑定 `TaskAnalysisState`，不再继续吃主 ViewModel 的这组平铺属性
- 预览区已经不再直接复用任务级 `CurrentFile`，而是回到“当前选中文件”语义
- `PreviewSelectionState` 已经开始承接 preview 子域的展示状态，`PreviewPanel.axaml` 的文件预览相关区域不再继续直接绑定一长串主 ViewModel 扁平 preview 属性
- `PreviewRenderCoordinator` 已经开始承接 `SelectedImage` 切换后的 preview render 时序，`MainWindowViewModel` 不再直接手拼“清理旧预览 -> 恢复 per-file 选择 -> 触发 GIF/PDF/PSD preview load”这一整段分支
- 体积预估已经补上 AI / 多页 PDF / 多帧 GIF 的免责声明
- 透明背景被压平成背景填充的规则，已经开始从 `ViewModel` 临时判断收口到 `ConversionPlan`

### 当前仍然故意没有动的高风险区

- 没有直接大拆 `MainWindowViewModel`
- 没有直接大拆 `ImagePreviewWindow`
- 没有全面移除 `AppServices`
- 没有重写整个转换执行主链
- 没有一次性删除旧模型上的全部历史字段

一句话总结当前状态：

> 现在还不是“已经整理完”，而是“已经把正确的推进边界、共享摘要语义和下一段真正该收口的主线打出来了”。

## 到目前为止已经推进了什么

下面只保留真正对后续接手有帮助的主线，不再展开成大段流水账。

### 已经完成的阶段性收口

- 状态边界已经立起来：`ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState`、`PreviewSessionState`、`ConversionPlan` 已经不是概念草图，而是实际开始承接职责。
- 旧设置兼容层已经立起来：`Services/AppSettingsStateMapper.cs` 已经把旧设置对象与新状态模型之间的映射收口，后续迁移不需要再回到“大而全”的旧对象上直接扩字段。
- 监听模式已经从“主界面瞬时状态的附属触发器”往“长期运行任务”推进：启用监听时固化 `ConversionJobDefinition` 快照，监听执行优先吃冻结规则，也支持显式“保存当前参数为监听配置”。
- AI 抠图已经开始回归预览工具语义：相关状态从批量转换 / 预设语义里逐步抽离，避免继续和主批处理管线混用。
- 轻量规划层已经开始接管解释性规则：`ConversionPlanningService` 已承接 AI 是否生效、哪些输入跳过 AI、GIF/PDF 展开、预计输出项、工作项、体积预估免责声明等说明性信息。
- 透明被压平为背景填充的规则，已经不再只是 `ViewModel` 里的局部判断，而是开始收口到 `ConversionPlan`，并被转换前预警、右侧任务分析、当前选中文件告警共同消费。
- 手动转换已经用 `RuntimeStatusSummary` 和 `CompletionSummaryModel` 收拢“运行中状态 / 完成后摘要 / 历史摘要”的共享语义。

### 我这一轮新增推进

- `PreviewRenderCoordinator` 这一轮继续往前吃下了 `PDF 页切换`、`GIF trim range`、`GIF specific frame` 三类 preview control 入口；这些入口不再让 `PdfViewModel`、`GifTrimRangeViewModel`、`GifSpecificFrameViewModel` 直接决定第一层 `RefreshSelectedPdfPreview()` / `LoadGifPreviewAsync()` / `ApplyGifSpecificFramePreview()` 的触发时机。
- `MainWindowPreviewRenderContext` 这轮补出了 `TryApplySelectedGifTrimPreviewRange()` 与 `TryApplySelectedGifSpecificFramePreview()` 两个明确入口，让 coordinator 可以先判断“已加载帧能否直接复用”，只在必要时才回落到重新加载 GIF preview，而不是继续把这套判断散落在各个 partial 里。
- `GifTrimRangeViewModel` 已从“自己判断当前有没有已加载帧、没有就自己发起 `LoadGifPreviewAsync()`”退到“更新选择 -> 交给 coordinator 决策”；GIF 裁剪区间的 preview reload 时序开始有统一出口。
- `GifSpecificFrameViewModel` 已从“自己决定直接套用已加载帧还是重新加载 GIF preview”退到“索引变化 -> 交给 coordinator 决策”；指定帧预览的第一层分流已经收口。
- `PdfViewModel` 里的页切换预览刷新与 PDF 解锁后的首次预览刷新，这轮也已经改为走 `PreviewRenderCoordinator`，避免 PDF preview control 继续保留一条独立直连。
- 本轮仍然没有新增 UI 文案键，因此没有扩散 10 语种维护面；后续若再新增文案键，仍要同步简中、繁中、英语、日语、韩语、德语、法语、意大利语、俄语、阿拉伯语这 10 种语言。
- 已补回归测试：`Tests/ImvixPro.Tests/ImageConversionTests.cs` 这轮新增钉住了 `PDF 页切换 -> coordinator`、`GIF trim range -> coordinator`、`GIF specific frame -> coordinator` 三条回归路径，避免后面再把 preview control 触发时机退回到 partial 直连。

### 当前仍未解掉的核心堵点

- `PreviewRenderCoordinator` 现在已经吃下 `selection + output + background + pdf page + gif trim + gif specific frame` 六组入口，但 `GIF handling mode change`、`specific-frame playback toggle`、`RestoreGifSpecificFrameSelection()` 里延迟回填的 preview 分支，以及少量 preview localization / state refresh 仍散落在 `GifSpecificFrameViewModel` 与 `PreviewViewModel`。
- `MainWindowShellCoordinator` 已经统一了 preview / detail / pdf unlock 的 request source，但 `MainWindow.axaml.cs` 仍然持有 tray icon 初始化/更新、窗口位置恢复、文件夹选择器、启动参数导入等宿主逻辑，还没有进一步退化为更薄的“平台事件转发 + host state / window chrome”层。
- `MainWindowViewModel` 和 `ImagePreviewWindow` 仍然是两个大宿主，边界虽然更清楚了，但结构风险还在。
- `AppServices` 仍然是运行期跨层入口，依赖关系还不够透明。
- `AppSettings` / `ConversionOptions` / `ConversionPreset` 仍然有兼容层噪音，不能再继续横向长字段。

## 下一任最建议从哪里继续推

### 先说结论

下一任不要再回头补 `history + notification`、`task analysis sidebar`，也不要再围着 `PreviewSelectionState` 或“shell coordinator 已经有了”做零碎修补了；这些边界和第一层协调器入口现在都已经立住了，继续在这里打转只会重新掉回“边角优化”。

更适合下一任接手的是一个比本轮更重、而且应该明确做成“子域抽离”的工作包：

> **剩余 GIF Preview Session 收口 + Host Shell 瘦身包**

目标不再是“把 output / background 再补一补”，而是顺着这轮已经推进出来的 `PreviewRenderCoordinator` 与 `MainWindowShellCoordinator` 继续前推，把剩余 `gif preview session` 分支和 main-window 宿主杂务一起收掉，避免下一轮又掉回小修小补。

### 下一任最推荐的中等偏重工作包

建议下一任至少同时完成下面五件事：

1. 沿着 `PreviewRenderCoordinator` 继续往前推，把 `GIF handling mode change`、`specific-frame playback toggle`、`RestoreGifSpecificFrameSelection()` 的延迟回填、以及剩余 preview localization / state refresh 触发链继续吃进去，目标是停止让 `GifSpecificFrameViewModel` / `PreviewViewModel` 继续直接判断“是立刻套用已加载帧，还是重新加载 preview”。
2. 上面一步做完后，必须顺手删除已经被 coordinator 吃掉的旧 helper / 重复出口，尤其是 `PreviewViewModel`、`GifSpecificFrameViewModel`、`GifTrimRangeViewModel`、`PdfViewModel` 里那批“coordinator 已经能管，但旧直连分支还在”的旧三轨。
3. 顺着 `MainWindowShellCoordinator` 再往前推一层，把 tray icon update、window placement restore/save、startup path import、文件夹选择器这些 main-window 宿主逻辑开始收进更薄的 host helper / shell 子域，让 `MainWindow.axaml.cs` 尽量只剩平台事件与窗口生命周期转发。
4. 如果前三步收得够稳，开始盘点 `ImagePreviewWindow` 内最先适合抽成 `tool controller` 的一类工具，优先挑互斥关系清楚、状态边界最容易抽的那一块做第一刀，但不要整窗大拆。
5. 收尾时必须继续做 `build + test + 启动 + 测试残留清理`，并检查这次没有引入新的文案键；若有新增 UI 文案，仍需同步 10 语种。

### 为什么我建议下一任把范围继续加大

- 这轮已经把 `selection + output + background + pdf page + gif trim + gif specific frame` 六组 preview refresh 入口收进了一层，再回头只改边角已经不会带来本质收益。
- 现在真正拖慢推进速度的，是剩余 `gif preview session` 仍然把 preview reload 时机散落在多个 partial 里，以及 `MainWindow.axaml.cs` 仍然承担过多宿主杂务；如果不一起往前推，项目会继续停在“coordinator 结构有了，但最难啃的会话态分支还留在旧壳里”。
- 只要下一任把剩余 GIF preview session 分支和 host shell 瘦身一起拿下一大块，后面无论是继续拆 `MainWindowViewModel`，还是开始碰 `ImagePreviewWindow` 的 tool controller，都会明显轻很多。
- 这个建议虽然比本轮更重，但仍然没有越线到“重写执行主链”或“直接大拆预览窗口”，风险依旧可控，而且比继续做小补丁更接近真正提速。

### 建议的实际落地顺序

第一步：

- 先把剩余仍会直接触发 preview reload 的入口按三组盘清：`gif handling/playback`、`gif restore/localization`、`host shell`，并明确哪些已经可以统一走 `PreviewRenderCoordinator`，哪些只是旧直连残留。

第二步：

- 把上面三组里最重的两组优先吃进 `PreviewRenderCoordinator` / host helper，每吃进一组就同步删掉一轮旧直连调用，不要只加 coordinator 不删旧 helper。

第三步：

- 再把 tray icon / window placement / startup path / folder picker 这批 main-window 宿主逻辑按“host state / shell helper / window chrome”拆清，至少先拿掉一整块 `MainWindow.axaml.cs` 里当前仍然偏业务化的代码。

第四步：

- 如果前三步收稳，就开始对 `ImagePreviewWindow` 做第一刀 tool controller 盘点或最小抽离；不要在这一步同时并行大动 `AppServices`。

第五步：

- 完成后必须做 `build + test + 启动 + 测试残留清理`。

### 如果下一任还有余力

- 可以顺手开始把 tray icon update / window placement 继续收进更薄的 host state 或 window chrome helper，但不要和 preview refresh 二次收口并行大拆。
- 可以顺手开始盘点 `AppServices` 在 main-window / preview path 上最先可替换的几个点，为后续 composition root 收缩做准备。
- 但下一轮仍然不建议直接大拆 `ImagePreviewWindow` 或直接全面迁移 `AppServices`，除非前面的 preview refresh 二次收口已经验证稳定。

## 一个实用判断标准

如果你新增或调整一条规则时，出现了下面任一情况：

- 需要同时改很多个 `ViewModel` 条件分支
- 需要同时补很多个 UI 入口说明
- 需要猜“手动转换”和“监听模式”是否表现一致
- 需要在预警、摘要、提示、历史里重复写相同逻辑

那基本说明这条规则还没有真正进入 `ConversionPlan / WatchProfile / SummaryCoordinator / Policy` 这一层。

这时正确动作通常不是继续堆分支，而是先收口，然后顺手删掉旧出口。

## 给下一任的最后一句话
