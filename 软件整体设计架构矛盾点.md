# 软件整体设计架构矛盾点

## 审查说明

- 本文基于当前代码仓库的静态审查整理，不修改现有代码，不引入测试文件或临时文件。
- 审查重点聚焦于 `Views`、`ViewModels`、`Services`、`Models` 之间的职责边界、状态流转方式、UI 语义一致性，以及 AI / GIF / PDF / 监听模式等复杂路径的协同关系。
- 总体结论：当前项目的核心矛盾并不是“功能少”，而是“当前任务配置、默认配置、监听任务配置、预览工具配置”被混放在同一套状态模型中，导致 UI 语义、执行逻辑、扩展方向三者逐步脱钩。

## 总体判断

- 当前项目已经具备商业化桌面工具的功能密度，但架构尚未形成稳定的“任务定义 -> 规则解析 -> 执行管线 -> 结果呈现”闭环。
- 目前大量能力依赖 `MainWindowViewModel` 和 `ImagePreviewWindow` 这两个中心对象协调，短期还能继续演进，长期会明显抬高功能冲突与回归风险。
- 最优先需要治理的不是“重写”，而是先把规则讲清楚、状态拆清楚、入口收敛清楚。

## 建议优先级

| 优先级 | 建议先处理的主题 | 原因 |
| --- | --- | --- |
| P0 | 区分“当前任务配置 / 默认偏好 / 监听任务配置 / 预览工具配置” | 这是大多数功能冲突和 UI 误导的共同根源 |
| P0 | 明确 AI 增强与传统压缩/缩放/输出的优先级规则 | 当前已有真实执行逻辑，但用户不可见，且估算与提示未同步 |
| P1 | 收敛重复状态展示面板，确定唯一主摘要出口 | 当前状态、摘要、历史在多个区域重复表达 |
| P1 | 为手动转换、监听转换、预览 AI 工具建立统一任务定义与策略解析层 | 当前三条路径规则不一致，后续扩展风险高 |
| P2 | 逐步拆分巨型 ViewModel / Window 和静态服务定位器 | 这是中长期维护成本的主要来源 |

## 一、功能冲突与执行规则不明确

### 1. AI 增强与传统压缩/缩放/输出格式的优先级是隐式规则，不是显式设计

- 问题描述：当前系统在 AI 增强开启时，会先执行 AI 增强，再把增强后的结果继续交给传统转换服务做缩放、压缩、格式编码，但界面没有明确说明这一规则，也没有解释冲突时的实际优先级。
- 触发场景：用户同时启用 `AI 增强 + 自定义缩放 + JPEG/WEBP 压缩 + 指定输出格式`。
- 影响范围：AI 增强、压缩、缩放、输出格式、预估体积、用户预期管理。
- 问题本质分析：`ConversionPipelineService` 只负责“AI 开/关”这一层包装，真正的规则分解仍分散在 `AiImageEnhancementService`、`ImageConversionService`、`ImageSaver` 中。系统已经形成了“AI 预处理 -> 常规转换”的既定事实，但没有形成用户可理解的策略说明。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：在 UI 和文档中明确写出当前规则为“AI 先执行，随后仍按当前压缩、缩放、输出格式继续处理”；在开始转换前给出规则摘要；对“AI 放大后又被缩小”的组合增加明确提示。
- 中长期推荐架构优化方向：引入 `Pipeline Stage` 或 `ProcessingPolicyResolver`，把 `Preprocess / Transform / Encode` 的优先级从代码隐式行为升级为可配置、可解释的策略层。
- 是否影响未来扩展能力：是。后续若加入 AI 降噪、AI 修复、AI 压缩等功能，冲突会指数级增加。
- 主要证据：`Services/ConversionPipelineService.cs`、`Services/AI/Enhancement/AiImageEnhancementService.cs`、`Services/ImageConversion/ImageConversionService.cs`、`Services/ImageConversion/ImageSaver.cs`。

### 2. AI 不支持的输入采用“跳过 AI、继续普通转换”的策略，但该策略没有被正式定义

- 问题描述：当批量任务中包含 PDF、GIF 或其他不适合 AI 增强的输入时，系统不是终止任务，也不是强制报错，而是隐式跳过 AI，继续执行普通转换。
- 触发场景：用户对混合批次开启 AI 增强，例如 `PNG + PDF`、`JPG + 动图 GIF`、`普通图片 + PSD/PDF` 混合导入。
- 影响范围：批处理一致性、用户预期、质量稳定性、异常诊断。
- 问题本质分析：当前只有“支持数量提示”和失败日志，没有“严格模式 / 跳过模式 / 转普通模式”的显式策略定义。结果是同一批次中的文件可能走完全不同的处理链，但界面上只显示为一次统一任务。
- 风险等级：中。
- 不进行大规模重构前提下的优化建议：在开始转换前增加“本批次中哪些文件将跳过 AI，哪些文件仍走普通转换”的说明；至少提供“跳过不支持项并继续”这一策略的文案确认。
- 中长期推荐架构优化方向：引入 `EligibilityPolicy` 和 `FallbackPolicy`，把“遇到不支持输入时如何处理”定义成统一策略，而不是散落在执行器内部。
- 是否影响未来扩展能力：是。后续只要增加新的 AI 能力，输入支持矩阵会更复杂。
- 主要证据：`Services/AI/Enhancement/AiImageEnhancementService.cs`、`Services/ConversionPipelineService.cs`、`ViewModels/Main/AiViewModel.cs`、`ViewModels/Main/ConversionViewModel.cs`。

### 3. 透明背景处理规则与 UI 文案并不完全一致

- 问题描述：界面上大量文案仍沿用“SVG 背景色”或“SVG 设置”，但实际执行时该背景填充规则不仅影响 SVG，还会影响透明 PSD、透明光栅图，且在输出到 JPEG / GIF / PDF 等不支持透明的格式时会被强制启用。
- 触发场景：用户把透明 PNG、PSD、SVG 转为 JPEG / GIF / PDF，或在预览中调整背景色。
- 影响范围：透明图输出、预览一致性、格式选择提示、UI 理解成本。
- 问题本质分析：底层规则实际已经是“背景填充策略”，但 UI 和模型命名仍残留“SVG 专用配置”的历史语义，导致同一开关在不同素材上承担了不同含义。
- 风险等级：中高。
- 不进行大规模重构前提下的优化建议：统一把相关文案收敛为“背景填充”；在透明素材转不透明格式时，补充明确风险提示，不再只对 JPEG 做单独提醒。
- 中长期推荐架构优化方向：把背景处理单独抽成 `TransparencyHandlingPolicy`，与 SVG、PSD、普通光栅源解耦。
- 是否影响未来扩展能力：是。后续新增更多输入格式或 AI 结果图时，当前命名会越来越误导。
- 主要证据：`ViewModels/Main/PsdViewModel.cs`、`ViewModels/Main/MainWindowViewModel.UiText.cs`、`Services/ImageConversion/ImageSaver.cs`、`Services/ImageAnalysisService.cs`。

### 4. GIF / PDF 的多帧多页规则在手动转换与监听模式中并不一致

- 问题描述：手动转换时，GIF/PDF 的当前页、全部页、页范围、全部帧等规则主要来自用户当前选择；监听模式下又会额外注入 `ApplyWatchContentAwareOptions`，强制把某些情况改成“全部页 / 全部帧”。
- 触发场景：同一套设置分别用于手动转换和监听转换，尤其是多页 PDF、多帧 GIF 输出图片时。
- 影响范围：监听模式稳定性、用户对自动化结果的可预测性、任务复现能力。
- 问题本质分析：监听模式并不是复用“同一任务定义”，而是在执行前再补一层特殊规则。这样会造成“同样的 UI 选择，在不同入口表现不同”。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：在监听模式设置区明确展示“监听模式附加规则”；提供单独的监听模式转换策略说明，而不是默认为与手动完全一致。
- 中长期推荐架构优化方向：把手动任务和监听任务都落到同一个 `JobDefinition` 上，再通过统一的 `ScenarioPolicyResolver` 决定默认行为。
- 是否影响未来扩展能力：是。未来只要增加更多输入特例，监听模式会越来越像“旁路系统”。
- 主要证据：`ViewModels/Main/ConversionViewModel.cs`、`ViewModels/Main/ConversionWatchViewModel.cs`、`Services/ImageConversion/ImageConversionService.cs`、`Services/ImageConversion/FormatConverter.cs`。

### 5. 体积预估与风险提示没有跟上真实执行链路

- 问题描述：当前体积预估主要按输出格式、压缩和缩放估算，但没有把 AI 增强导致的分辨率变化、PDF 多页导出、GIF 多帧拆分等关键因素完整纳入。
- 触发场景：开启 AI 增强、PDF 导出全部页、GIF 导出全部帧、GIF 转 PDF 等。
- 影响范围：右侧预估信息、历史记录中的预估值、用户对磁盘占用和耗时的判断。
- 问题本质分析：分析层仍按“单输入 -> 单输出”思路估算，而执行层已经支持“单输入 -> 多输出”与“先增强再编码”的复杂路径，导致估算模型与执行模型脱节。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：先把 AI 增强比例、多页 PDF、多帧 GIF 纳入估算；若暂时无法准确计算，应明确显示“估算不含 AI 放大 / 多页拆分”的免责声明。
- 中长期推荐架构优化方向：将估算逻辑放到 Pipeline 规划阶段，根据实际 `WorkPlan` 生成预估，而不是直接对原始图片做静态估算。
- 是否影响未来扩展能力：是。任何新的预处理或拆分能力都会继续放大误差。
- 主要证据：`Services/ImageAnalysisService.cs`、`ViewModels/Main/ConversionViewModel.cs`、`Services/ImageConversion/ImageConversionService.cs`、`Services/ImageConversion/FormatConverter.cs`。

### 6. AI 批次执行中的 GPU / 模型回退会影响同批后续文件，但该行为对用户不可见

- 问题描述：当前 AI 增强一旦在某个文件上触发 GPU 回退或默认模型回退，后续文件可能直接沿用回退后的硬件或模型策略。
- 触发场景：批处理过程中某个文件触发 GPU 失败、模型加载失败或环境兼容性问题。
- 影响范围：同批次结果一致性、性能稳定性、问题复现。
- 问题本质分析：当前执行器把“回退策略”做成了批次内共享状态，这是工程上合理的容错手段，但没有在任务层把它定义为用户可理解的运行策略。
- 风险等级：中。
- 不进行大规模重构前提下的优化建议：在任务摘要或日志中记录“本批次发生 GPU 回退 / 模型回退，并对后续文件生效”；界面增加一次性提示。
- 中长期推荐架构优化方向：把硬件选择、模型回退、容错级别提升为 `ExecutionPolicy`，由调度层统一管理与记录。
- 是否影响未来扩展能力：是。未来一旦有多种 AI 引擎并存，这类隐式回退会更难追踪。
- 主要证据：`Services/AI/Enhancement/AiImageEnhancementService.cs`、`ViewModels/Main/AiViewModel.cs`。

## 二、UI 冗余与语义误导

### 7. “默认设置”与“当前任务设置”共用同一组属性，UI 语义存在误导

- 问题描述：顶部工具栏和设置面板同时编辑 `SelectedOutputFormat`、`OutputDirectory` 等同一组状态；设置页文案却写成“默认输出格式”“默认输出目录”，容易让用户误以为这里修改的是未来默认值而不是当前任务。
- 触发场景：用户在顶部修改输出格式后，再进入设置页；或在设置页修改“默认值”后立即开始当前转换。
- 影响范围：当前转换行为、下次启动默认值、监听模式配置来源、用户心智模型。
- 问题本质分析：系统没有真正区分“当前任务快照”和“应用默认偏好”。同一套 UI 状态既代表正在编辑的任务，又代表会被立即持久化的默认设置。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：先统一文案，把设置页改成“当前转换设置”；或明确标注“修改会立即影响当前任务并保存为默认值”；减少一处重复入口，保留单一主编辑面。
- 中长期推荐架构优化方向：拆分 `AppPreferences` 与 `ConversionProfile`，并为“保存为默认”提供显式动作，而不是属性变更即落盘。
- 是否影响未来扩展能力：是。后续只要引入模板、队列、监听任务、批量方案，这种混用都会成为核心阻塞点。
- 主要证据：`Views/Controls/TopToolbar.axaml`、`Views/Controls/SettingsPanel.axaml`、`ViewModels/Main/MainWindowViewModel.Configuration.cs`、`Models/AppSettings.cs`。

### 8. 状态、摘要、历史在多个区域重复表达，且上下文边界不清

- 问题描述：当前系统同时在右侧预览区、底部状态栏、AI 反馈条、完成弹窗、历史卡片中表达任务状态或摘要信息；其中 `CurrentFile` 甚至同时承担“当前处理文件”和“完成后的摘要文本”两种含义。
- 触发场景：执行转换时、转换完成后、查看右侧预览区时。
- 影响范围：主界面信息密度、状态可读性、认知负担、国际化文案维护。
- 问题本质分析：系统缺少“实时状态 / 一次性摘要 / 持久化历史”的信息分层设计，导致同一数据在多个区域被重复消费，而且每个区域都想承担完整说明责任。
- 风险等级：中。
- 不进行大规模重构前提下的优化建议：确定唯一主状态区域和唯一完成摘要区域；预览区只保留与所选文件直接相关的信息；不要再在预览区复用任务级 `CurrentFile`。
- 中长期推荐架构优化方向：建立 `RuntimeStatusModel`、`CompletionSummaryModel`、`HistoryEntryModel` 三层展示模型，禁止同一字段跨多个语义层复用。
- 是否影响未来扩展能力：是。功能越多，重复信息面板越难维护，文案冲突也会更多。
- 主要证据：`Views/MainWindow.axaml`、`Views/Controls/PreviewPanel.axaml`、`Views/ConversionSummaryWindow.axaml`、`Views/MainWindow.axaml.cs`、`ViewModels/Main/ConversionViewModel.cs`、`ViewModels/Main/FileListViewModel.cs`。

### 9. 右侧“预览”标签页实际上同时承担预览、规则编辑、风险提示、失败日志、历史浏览五种职责

- 问题描述：预览面板不仅显示预览图，还承载 PDF/GIF 规则设置、背景填充设置、警告、估算、失败日志、历史记录等多种职责。
- 触发场景：选中任意文件并切到预览页。
- 影响范围：信息架构、界面扫描成本、用户操作路径。
- 问题本质分析：系统把“与当前文件相关的信息”和“与整个任务相关的信息”混在同一标签页内，导致标签名称与实际职责不匹配，也让用户难以判断某项配置是否只作用于当前文件。
- 风险等级：中。
- 不进行大规模重构前提下的优化建议：先按语义把预览区内容分成“文件预览 / 本次任务分析 / 历史与故障”三组；至少把历史和失败日志移出预览主区域。
- 中长期推荐架构优化方向：将右侧区域拆成 `Preview`、`Task Analysis`、`History/Diagnostics` 三个明确子域，或采用 feature-specific panel coordinator。
- 是否影响未来扩展能力：是。继续往预览页塞功能会快速失控。
- 主要证据：`Views/Controls/PreviewPanel.axaml`。

### 10. AI 抠图被放进全局 AI 面板和预设体系，但真实能力只存在于预览窗口

- 问题描述：主界面的 AI 面板提供了完整的抠图模型、设备、输出格式、背景、边缘优化、分辨率策略设置；预设和设置也会保存这些值，但主转换流程并不会使用它们。
- 触发场景：用户在 AI 面板配置抠图，保存预设，随后执行批量转换。
- 影响范围：AI 功能理解、预设含义、设置可信度、后续扩展边界。
- 问题本质分析：抠图当前本质上是“预览窗口里的交互式工具”，不是主批处理 Pipeline 的阶段，但它已经被放入了全局任务配置模型，形成了强烈的语义错位。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：在主界面明确标注“预览工具专用，不参与批量转换”；不要把它与批量 AI 增强并列成同等级的转换设置。
- 中长期推荐架构优化方向：把抠图配置从 `ConversionOptions` / `ConversionPreset` 中抽离，改为 `PreviewToolOptions` 或 `AiMattingToolState`。
- 是否影响未来扩展能力：是。未来只要把抠图真正接入批处理，就会立即撞上现有命名和数据模型的历史包袱。
- 主要证据：`Views/Controls/AiPanel.axaml`、`ViewModels/Main/AiViewModel.cs`、`ViewModels/Main/ConversionViewModel.cs`、`Models/ConversionOptions.cs`、`Models/ConversionPreset.cs`、`Views/ImagePreviewWindow.Matting.cs`。

### 11. 模型层存在未完成或无效配置，增加理解噪音

- 问题描述：如 `AiPanelEnabled` 被保存到设置和任务选项中，但实际被强制保持为 `true`；`OutputDirectoryRule.AutoCreateResultFolder` 和 `AutoResultFolderName` 在模型与执行层存在，但当前主界面没有对应入口。
- 触发场景：阅读代码、维护配置兼容、扩展设置面板时。
- 影响范围：模型可信度、维护效率、后续功能扩展。
- 问题本质分析：模型层保留了部分历史或预留能力，但这些能力并没有完整走通到 UI 和行为层，导致“看起来支持，实际没有完整产品语义”。
- 风险等级：低中。
- 不进行大规模重构前提下的优化建议：在架构文档中明确哪些配置是保留字段、哪些是当前正式可用能力；避免继续把未走通的字段扩散到更多层。
- 中长期推荐架构优化方向：建立配置项生命周期管理，区分“正式支持 / 内部预留 / 迁移兼容”。
- 是否影响未来扩展能力：是，但属于中低强度影响。主要问题是噪音和误判。
- 主要证据：`ViewModels/Main/AiViewModel.cs`、`Models/ConversionOptions.cs`、`Models/AppSettings.cs`、`Models/ConversionPreset.cs`、`Models/OutputDirectoryRule.cs`、`Services/ImageConversion/ImageConversionService.cs`。

## 三、架构设计缺陷与扩展性风险

### 12. MainWindowViewModel 以“分文件”伪装成“分模块”，实质仍是单一巨型 ViewModel

- 问题描述：`ConversionViewModel.cs`、`FileListViewModel.cs`、`AiViewModel.cs`、`PdfViewModel.cs`、`PreviewViewModel.cs`、`ConversionWatchViewModel.cs` 等多个文件名看起来像独立 ViewModel，实际全部都是 `MainWindowViewModel` 的 partial。
- 触发场景：新增功能、定位 Bug、交接维护、单元测试。
- 影响范围：主界面全部功能、状态同步、属性变更链、命令依赖、可测试性。
- 问题本质分析：当前只是把“大类”拆成了多个文件，而没有形成真正的运行时边界。所有模块依然共享同一对象、同一生命周期、同一属性空间。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：先为 `MainWindowViewModel` 明确子域边界文档，约束新增逻辑进入对应 partial；减少跨 partial 直接读写字段；逐步引入 coordinator 或 sub-state 类。
- 中长期推荐架构优化方向：将主界面拆成 `FileListState`、`ConversionState`、`AiState`、`PreviewState`、`WatchState` 等独立 ViewModel / Store，并由更薄的壳层协调。
- 是否影响未来扩展能力：是。这是当前主界面长期演进的最大结构性风险之一。
- 主要证据：`ViewModels/Main/*.cs` 中多处 `public partial class MainWindowViewModel`。

### 13. ImagePreviewWindow 也在演变为第二个巨型功能宿主

- 问题描述：预览窗口内部同时承载 AI 增强预览、AI 抠图、OCR、二维码、条码等多种交互式工具，并通过大量互斥标志位互相协调。
- 触发场景：在预览窗口继续增加任何新工具或新模式。
- 影响范围：预览交互稳定性、代码复杂度、工具之间的互斥关系。
- 问题本质分析：当前预览窗口缺少“工具插件宿主”或“预览会话管理器”，于是每新增一种能力，就会在同一个窗口对象上继续叠加状态位和互斥判断。
- 风险等级：中高。
- 不进行大规模重构前提下的优化建议：先把各工具的 busy 状态、缓存、保存逻辑封装成局部 coordinator；减少窗口根类直接持有的布尔状态数量。
- 中长期推荐架构优化方向：把预览窗口改造成工具宿主，按工具注册 `PreviewToolController`，统一处理启停、互斥、缓存和 UI 呈现。
- 是否影响未来扩展能力：是。预览工具越多，当前互斥逻辑越难维护。
- 主要证据：`Views/ImagePreviewWindow.axaml.cs`、`Views/ImagePreviewWindow.Ai.cs`、`Views/ImagePreviewWindow.Matting.cs`、`Views/ImagePreviewWindow.Qr.cs`、`Views/ImagePreviewWindow.Barcode.cs`。

### 14. ConversionOptions / AppSettings / ConversionPreset 三个模型同时承担了过多职责

- 问题描述：这三类对象同时混合了当前转换参数、默认偏好、预览工具参数、监听模式共享参数、按文件缓存选择等多种职责。
- 触发场景：保存设置、应用预设、执行监听模式、打开预览工具、未来新增处理阶段。
- 影响范围：配置一致性、序列化兼容、功能扩展、调试定位。
- 问题本质分析：系统希望用“一套大对象”贯穿所有场景，但执行层又只消费其中一部分。例如主界面会组装完整 AI / PDF / GIF / 抠图字段，`ImageSaver.NormalizeOptions` 却只保留传统转换子集，说明模型表面统一、语义上并不统一。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：先定义字段归属表，明确哪些属于 `AppPreferences`、哪些属于 `JobOptions`、哪些属于 `PreviewToolOptions`；停止继续向这三个对象横向加字段。
- 中长期推荐架构优化方向：分离 `ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState`、`PerItemSelectionState`。
- 是否影响未来扩展能力：是。这类“万能配置对象”会直接成为长期维护瓶颈。
- 主要证据：`Models/AppSettings.cs`、`Models/ConversionOptions.cs`、`Models/ConversionPreset.cs`、`ViewModels/Main/ConversionViewModel.cs`、`Services/ImageConversion/ImageSaver.cs`。

### 15. 缺少统一的 Pipeline / Policy / Scheduler 抽象，导致多入口逻辑各自生长

- 问题描述：当前项目虽然已有 `ConversionPipelineService`，但它只处理 AI 增强开关，并没有真正承担统一任务规划职责；AI 抠图走预览窗口旁路，监听模式有独立附加规则，PDF/GIF 特例则散落在转换服务和 ViewModel 中。
- 触发场景：新增一种预处理、后处理、导出模式或自动化入口。
- 影响范围：执行链一致性、规则可解释性、功能扩展成本。
- 问题本质分析：目前更像“若干服务 + 若干特殊分支”的组合，而不是“统一任务模型驱动的执行管线”。这会让同一需求在不同入口出现不同实现。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：先引入轻量的 `Plan` 概念，在执行前统一生成“本任务将包含哪些阶段、哪些输入会被拆分、哪些输入会被跳过 AI”的可视化计划。
- 中长期推荐架构优化方向：建立正式的 `Pipeline`、`PolicyResolver`、`TaskScheduler` 三层结构，统一承接手动转换、监听转换、预览导出、AI 工具保存等路径。
- 是否影响未来扩展能力：是。这是项目能否稳定进入长期商业维护阶段的关键。
- 主要证据：`Services/ConversionPipelineService.cs`、`ViewModels/Main/ConversionViewModel.cs`、`ViewModels/Main/ConversionWatchViewModel.cs`、`Views/ImagePreviewWindow.Ai.cs`、`Views/ImagePreviewWindow.Matting.cs`、`Services/ImageConversion/FormatConverter.cs`。

### 16. AppServices 静态服务定位器贯穿多层，导致依赖关系隐藏

- 问题描述：项目表面上存在构造函数注入，但大量底层服务、帮助类、模型创建路径仍直接通过 `AppServices` 获取全局依赖。
- 触发场景：替换实现、编写测试、引入多实例上下文、调整生命周期管理。
- 影响范围：服务解耦、测试隔离、运行时可控性、未来模块化。
- 问题本质分析：当前是“部分依赖注入 + 部分全局定位器”的混合模式，导致真实依赖链并不透明，很多底层代码看似纯函数，实际依赖全局单例。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：先停止新增 `AppServices` 直接调用；对新增代码要求通过构造参数或局部工厂传入依赖；为关键路径补一份依赖图。
- 中长期推荐架构优化方向：建立单一 composition root，把 `AppServices` 逐步退化为启动阶段装配器，而不是运行时全局依赖入口。
- 是否影响未来扩展能力：是。尤其会阻碍插件化、并行任务上下文和更细粒度测试。
- 主要证据：`Services/AppServices.cs` 以及 `Services`、`ViewModels`、`Views`、`Models` 中大量 `AppServices.` 调用。

### 17. 监听模式缺少独立的转换档案，自动化能力会持续受手动操作影响

- 问题描述：监听模式目前只单独维护监听目录和开关，不维护独立的输出格式、压缩、AI、重命名、透明处理等完整任务档案。
- 触发场景：用户先配置监听模式，再临时修改主界面的手动转换参数。
- 影响范围：后台自动处理的稳定性、业务可追溯性、自动化场景可信度。
- 问题本质分析：监听模式本质应是“长期运行任务”，但当前只是“复用主界面当前状态的一种触发方式”，因此天然不稳定。
- 风险等级：高。
- 不进行大规模重构前提下的优化建议：增加“将当前参数保存为监听配置”的显式动作；至少在启用监听时固化一份快照，并允许用户查看当前监听使用的实际规则。
- 中长期推荐架构优化方向：为监听模式建立独立的 `WatchProfile`、`WatchJobDefinition` 与执行日志模型。
- 是否影响未来扩展能力：是。未来若需要多个监听任务、计划任务或自动化工作流，当前设计几乎无法平滑扩展。
- 主要证据：`Models/AppSettings.cs`、`ViewModels/Main/ConversionWatchViewModel.cs`、`ViewModels/Main/ConversionViewModel.cs`。

## 四、综合优化建议

### 短期建议：在不做大规模重构前提下

- 先明确四类状态边界：`当前任务`、`默认偏好`、`监听任务`、`预览工具`。
- 对 AI 增强、透明背景、多页 PDF、多帧 GIF 建立一份“规则优先级说明表”，并在 UI 中给出最少量但明确的提示。
- 收敛重复信息出口：底部状态栏只负责实时状态，完成摘要只保留一个出口，历史只保留持久记录，预览区只保留与当前选中文件直接相关的内容。
- 把 AI 抠图明确标注为“预览工具”，不要继续让它和批量转换参数混在同一语义层。
- 对监听模式增加“当前使用的转换规则摘要”，避免后台行为成为黑箱。
- 暂停继续向 `MainWindowViewModel`、`ConversionOptions`、`AppSettings` 横向追加新字段，优先先做边界梳理。

### 中长期建议：面向长期商业维护的架构方向

- 引入 `ApplicationPreferences`、`ConversionJobDefinition`、`WatchProfile`、`PreviewToolState` 四类独立模型。
- 引入统一 `Pipeline`，至少拆成 `Validate -> Analyze -> Preprocess -> Transform -> Encode -> Emit -> PostAction` 七个阶段。
- 引入 `PolicyResolver`，统一决定 AI 回退、透明处理、页帧展开、命名冲突、输出目录规则。
- 引入统一 `TaskScheduler`，承接手动任务、监听任务、预览导出、未来计划任务。
- 逐步把主界面拆成多个 feature-specific ViewModel / coordinator，而不是继续扩大 partial class。
- 逐步移除 `AppServices` 在运行期的直接调用，把依赖收口到启动装配层。

## 五、对未来扩展能力的总体影响

- 若不处理上述矛盾，后续每增加一个功能，都会同时触碰 UI 文案、配置模型、执行规则、预览入口和监听模式，回归风险会持续上升。
- 若优先完成“状态拆分 + 规则显式化 + 管线统一化”，则本项目非常适合继续扩展 AI 增强、AI 修复、批量水印、计划任务、更多文档格式导出等长期商业能力。
- 因此，本项目当前最关键的不是“重写”，而是先把架构边界讲清楚、规则层抽出来、入口路径统一起来。
